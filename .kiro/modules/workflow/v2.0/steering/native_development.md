# VibeCoding 原生功能开发流程

## 🎯 流程说明

本流程适用于从零开始开发新功能，包含 4 个核心阶段：
1. **Phase 1**：了解项目与需求（快速对齐）
2. **Phase 2**：分析需求（主动澄清）
3. **Phase 3**：制定执行计划（改测循环）
4. **Phase 4**：落地执行（小步快跑）
5. **Phase 5**：总结（可选）

**核心理念**：
- 轻量化，极简输出
- 改一点测一点
- 主动澄清所有不确定点
- 测试和文档可选

---

## 📋 Phase 1：了解项目与需求

### 核心动作

1. **确认需求边界**
   - 明确功能要实现的核心目标
   - 明确输入 / 输出
   - 明确与 kiro 现有模块的关联
     - 是否依赖 current_context.yaml 中的某个状态？
     - 是否复用 GUI 的某个组件？
     - 是否需要创建新的目录结构？

2. **快速扫查 kiro 现有资产**
   - 确认是否有可复用的代码片段
   - 确认是否有可复用的配置
   - 确认是否有可复用的 GUI 组件
   - 避免重复开发

3. **创建需求目录结构**（如果需要）
   - 询问用户：模块名称（kebab-case 格式）
   - 询问用户：版本号（如 v1.0）
   - 询问用户：需求主题（简短描述）
   - 创建目录：`requirements/{module}/{version}/`
   - 生成文件：`meta.yaml` 和 `requirement.md`
   - 更新文件：`current_context.yaml`

### 输出物（极简）

```
✅ Phase 1 完成

需求记录：
- 核心目标：[1-3 行文字描述]
- 输入/输出：[简要说明]
- 关联模块：[列出相关模块]

复用资产清单：
- [可复用的代码片段/配置/组件]
- [如果没有，写"无"]

目录结构：
- 已创建：requirements/{module}/{version}/
- 已生成：meta.yaml, requirement.md
- 已更新：current_context.yaml

待确认项：无

接下来进入 Phase 2：分析需求
```

---

## 📋 Phase 2：分析需求

### 核心动作

1. **拆解需求为最小可执行单元**
   - 将功能拆分为"改一点就能测"的小模块
   - 示例：
     - 先实现 GUI 按钮渲染
     - 再实现按钮点击事件
     - 再实现事件触发的状态更新

2. **主动澄清所有不确定点**（强制执行）
   
   **必问问题**：
   - ❓ 功能与 kiro 现有项目是否有冲突点？
   
   **条件问题**（根据需求判断）：
   - ❓ 如果涉及 GUI：GUI 交互细节偏好是什么？
     - 弹窗样式？
     - 交互触发方式？
     - 布局位置？
   
   - ❓ 是否需要测试？
     - A. 需要测试（请说明测试复杂度：简单/中等/复杂）
     - B. 不需要测试（跳过 Phase 3 和 Phase 4 的测试步骤）
   
   - ❓ 是否需要生成总结文档？
     - A. 需要（Phase 5 生成总结文档）
     - B. 不需要（跳过 Phase 5）

3. **记录用户的选择**
   - 测试需求：[需要/不需要]
   - 文档需求：[需要/不需要]
   - GUI 偏好：[具体偏好]
   - 其他澄清：[记录所有澄清的内容]

### 输出物（极简）

```
✅ Phase 2 完成

需求拆解：
1. [最小可执行单元 1]
2. [最小可执行单元 2]
3. [最小可执行单元 3]
...

用户选择：
- 测试需求：[需要/不需要]
- 测试复杂度：[简单/中等/复杂/无]
- 文档需求：[需要/不需要]
- GUI 偏好：[具体偏好/无]

澄清记录：
- [所有澄清的内容]

待确认项：无

接下来进入 Phase 3：制定执行计划
```

---

## 📋 Phase 3：制定执行计划

### 核心动作

1. **按"改一点测一点"拆分执行步骤**
   - 每个步骤仅包含 1 个最小可测试单元
   - 示例：
     - 步骤 1：在 gui/ 目录新增按钮组件 → 测试：渲染是否正常
     - 步骤 2：绑定按钮点击事件 → 测试：事件是否触发
     - 步骤 3：实现事件处理逻辑 → 测试：逻辑是否正确

2. **对齐 kiro 规范**
   - 每个步骤明确对应的目录
   - 每个步骤明确对应的状态管理点
   - 每个步骤明确对应的 GUI 交互逻辑

3. **根据 Phase 2 的选择调整计划**
   - 如果不需要测试：移除所有测试步骤
   - 如果不需要文档：移除 Phase 5

### 输出物（极简）

```
✅ Phase 3 完成

执行计划：

步骤 1：[最小单元 1]
- 修改文件：[文件路径]
- 修改内容：[简要说明]
- 测试点：[测试内容]（如果不需要测试，写"无"）

步骤 2：[最小单元 2]
- 修改文件：[文件路径]
- 修改内容：[简要说明]
- 测试点：[测试内容]（如果不需要测试，写"无"）

步骤 3：[最小单元 3]
- 修改文件：[文件路径]
- 修改内容：[简要说明]
- 测试点：[测试内容]（如果不需要测试，写"无"）

...

全流程测试：[如果需要测试，说明全流程测试内容；如果不需要，写"无"]

待确认项：无

接下来进入 Phase 4：落地执行
```

---

## 📋 Phase 4：落地执行

### 核心动作

1. **按计划执行单个最小单元开发**
   - 仅修改目标文件（如仅改 gui/Button.js、state/xxxState.js）
   - 不要一次性修改多个文件

2. **开发完成后立即测试**（如果 Phase 2 选择需要测试）
   - 仅测试当前修改的单元
   - 不做全量测试
   - 测试方式：
     - 简单测试：手动验证功能是否正常
     - 中等测试：编写简单的测试用例
     - 复杂测试：编写完整的测试套件

3. **测试结果处理**
   - 测试通过 → 推进下一个单元
   - 测试不通过 → 仅回滚当前修改，修复后重新测试
   - 不影响已完成的模块

4. **所有单元完成后，做一次轻量全流程测试**（如果需要测试）
   - 仅覆盖当前功能链路
   - 不做全系统测试

### 执行流程

```
开始执行步骤 1...
[修改文件]
[如果需要测试] → 测试步骤 1...
[测试结果]
✅ 步骤 1 完成

开始执行步骤 2...
[修改文件]
[如果需要测试] → 测试步骤 2...
[测试结果]
✅ 步骤 2 完成

...

[如果需要测试] → 全流程测试...
[测试结果]
✅ 全流程测试通过

✅ Phase 4 完成

所有功能已实现并测试通过！

接下来进入 Phase 5：总结（如果需要）
```

## 📋 Phase 5：总结（可选）

### 触发条件

- 如果 Phase 2 中用户选择"需要生成总结文档" → 执行 Phase 5
- 如果 Phase 2 中用户选择"不需要生成总结文档" → 跳过 Phase 5

### 核心动作（如果执行）

1. **生成总结文档**
   - 文件路径：`requirements/{module}/{version}/summary.md`
   - 内容包括：
     - 需求概述
     - 实现方案
     - 关键决策
     - 遇到的问题和解决方案
     - 可复用的经验

2. **更新 current_context.yaml**
   - 标记需求为已完成
   - 记录完成时间

### 输出物（如果执行）

```
✅ Phase 5 完成

已生成总结文档：
- 文件：requirements/{module}/{version}/summary.md
- 内容：需求概述、实现方案、关键决策、问题解决、可复用经验

已更新状态：
- current_context.yaml：标记需求为已完成

🎉 原生功能开发流程全部完成！
```

### 输出物（如果跳过）

```
⏭️ Phase 5 已跳过（用户选择不生成总结文档）

🎉 原生功能开发流程全部完成！
```

## 📊 流程图

```
Phase 1：了解项目与需求
    ↓
    - 确认需求边界
    - 扫查可复用资产
    - 创建目录结构
    ↓
Phase 2：分析需求
    ↓
    - 拆解为最小单元
    - 主动澄清不确定点
    - 询问测试和文档需求
    ↓
Phase 3：制定执行计划
    ↓
    - 按"改测"拆分步骤
    - 对齐 kiro 规范
    - 根据选择调整计划
    ↓
Phase 4：落地执行
    ↓
    - 执行单个最小单元
    - 立即测试（如果需要）
    - 测试通过→下一个
    - 测试不通过→回滚修复
    ↓
Phase 5：总结（可选）
    ↓
    - 生成总结文档（如果需要）
    - 更新状态
    ↓
完成
```

---

## 🎯 执行约束（强制遵守）

### 约束 1：极简输出
每个 Phase 的输出必须极简，不要冗长的文档。

### 约束 2：主动提问
Phase 2 必须主动澄清所有不确定点，不要猜测。

### 约束 3：改一点测一点
Phase 4 必须按最小单元执行，不要一次性修改多个文件。

### 约束 4：尊重用户选择
如果用户选择不需要测试或文档，必须严格遵守，不要强制执行。

### 约束 5：状态管理
必须正确维护 current_context.yaml 和 meta.yaml。
