# VibeCoding 二次开发流程

## 🎯 流程说明

本流程适用于基于现有代码进行改造，包含 5 个核心阶段：
1. **Phase 0**：评估相似代码（适配性分析）
2. **Phase 1**：了解项目与需求（快速对齐）
3. **Phase 2**：分析需求（主动澄清）
4. **Phase 3**：制定执行计划（改测循环）
5. **Phase 4**：落地执行（小步快跑）
6. **Phase 5**：总结（可选）

**核心理念**：
- 先评估，再开发
- 分析适配性、改造成本、风险
- 复用原生开发流程（Phase 1-4）
- 改一点测一点

---

## 📋 Phase 0：评估相似代码

### 核心动作

1. **确认相似代码的位置**
   - 询问用户：相似代码在哪里？
     - 已拷贝到当前项目根目录？
     - 在其他项目中？
     - 在 GitHub 仓库中？
   - 如果未拷贝，提示用户先拷贝到根目录

2. **分析相似代码与 kiro 项目的适配性**
   - **目录规范兼容性**：
     - 代码是否符合 kiro 的目录结构？
     - 是否需要重新组织文件？
   
   - **状态管理兼容性**：
     - 代码是否使用了状态管理？
     - 是否与 current_context.yaml 兼容？
     - 是否需要改造状态管理逻辑？
   
   - **GUI 逻辑兼容性**：
     - 代码是否包含 GUI 组件？
     - GUI 组件是否与 kiro 的 GUI 框架兼容？
     - 是否需要重写 GUI 部分？
   
   - **依赖兼容性**：
     - 代码依赖的库是否与 kiro 项目兼容？
     - 是否需要安装新的依赖？

3. **评估改造成本**
   - **低成本**：仅需少量修改（< 20% 代码）
   - **中成本**：需要中等修改（20%-50% 代码）
   - **高成本**：需要大量修改（> 50% 代码）
   - **建议重写**：改造成本高于重写成本

4. **提取可复用片段**
   - 标记可直接复用的部分（无需修改）
   - 标记少量改造的部分（< 10 行修改）
   - 标记需要重写的部分

5. **确认改造风险**
   - 是否存在与 kiro 现有代码的冲突？
     - 变量名重复？
     - 函数名重复？
     - 状态逻辑冲突？
   - 是否存在技术风险？
     - 不熟悉的技术栈？
     - 复杂的业务逻辑？

### 输出物（极简）

```
✅ Phase 0 完成

相似代码位置：
- [代码路径]

适配性分析：
- 目录规范：[兼容/需改造/不兼容]
- 状态管理：[兼容/需改造/不兼容]
- GUI 逻辑：[兼容/需改造/不兼容/无 GUI]
- 依赖兼容：[兼容/需安装新依赖]

改造成本评估：
- 成本等级：[低/中/高/建议重写]
- 预计改造代码量：[X%]
- 预计改造时间：[X 小时]

可复用片段：
- 直接复用：[列出可直接复用的文件/函数]
- 少量改造：[列出需少量改造的部分]
- 需要重写：[列出需要重写的部分]

改造风险：
- 冲突风险：[列出可能的冲突点]
- 技术风险：[列出技术风险]

建议：
- [是否建议二次开发？还是建议重写？]
- [如果建议二次开发，给出改造策略]

待确认项：
- ❓ 是否继续二次开发？还是改为原生开发（重写）？

请确认：
A. 继续二次开发（执行 Phase 1-4）
B. 改为原生开发（切换到原生开发流程）
C. 取消开发

请选择：A / B / C
```

### 用户选择处理

- **如果用户选择 A**：继续执行 Phase 1
- **如果用户选择 B**：切换到原生开发流程（`native_development.md`）
- **如果用户选择 C**：取消开发，正常作答

## 📋 Phase 1：了解项目与需求

### 核心动作

与原生开发流程的 Phase 1 相同，但需要额外：
- 将 Phase 0 的"相似代码复用计划"加入"复用资产清单"

### 输出物（极简）

```
✅ Phase 1 完成

需求记录：

- 输入/输出：[简要说明]
- 关联模块：[列出相关模块]

- [kiro 现有的可复用资产]
- [相似代码的可复用片段]（来自 Phase 0）

目录结构：
- 已创建：requirements/{module}/{version}/
- 已生成：meta.yaml, requirement.md
- 已更新：current_context.yaml

待确认项：无

接下来进入 Phase 2：分析需求
```

## 📋 Phase 2：分析需求

### 核心动作

与原生开发流程的 Phase 2 相同，但需要额外：
- 将"相似代码改造"作为需求的一部分进行拆解

### 输出物（极简）

```
✅ Phase 2 完成

需求拆解：
1. [最小可执行单元 1]
2. [相似代码改造单元 1]（标注为"改造"）
3. [最小可执行单元 2]
4. [相似代码改造单元 2]（标注为"改造"）
...

用户选择：
- 测试需求：[需要/不需要]
- 测试复杂度：[简单/中等/复杂/无]
- 文档需求：[需要/不需要]
- GUI 偏好：[具体偏好/无]

澄清记录：
- [所有澄清的内容]

待确认项：无

接下来进入 Phase 3：制定执行计划
```

## 📋 Phase 3：制定执行计划

### 核心动作

与原生开发流程的 Phase 3 相同，但需要额外：
- 将"相似代码改造"拆分为最小可测试单元
- 每个改造单元都要明确：
  - 改造前的代码
  - 改造后的代码
  - 改造原因（为什么要改）

### 输出物（极简）

```
✅ Phase 3 完成

执行计划：

步骤 1：[最小单元 1]
- 修改文件：[文件路径]
- 修改内容：[简要说明]
- 测试点：[测试内容]（如果不需要测试，写"无"）

步骤 2：[相似代码改造单元 1]（改造）
- 改造文件：[文件路径]
- 改造前：[原代码逻辑]
- 改造后：[新代码逻辑]
- 改造原因：[为什么要改]
- 测试点：[兼容性测试 + 功能测试]（如果不需要测试，写"无"）

步骤 3：[最小单元 2]
- 修改文件：[文件路径]
- 修改内容：[简要说明]
- 测试点：[测试内容]（如果不需要测试，写"无"）

...

全流程测试：[如果需要测试，说明全流程测试内容；如果不需要，写"无"]

待确认项：无

接下来进入 Phase 4：落地执行
```

## 📋 Phase 4：落地执行

### 核心动作

与原生开发流程的 Phase 4 相同，但需要额外：
- 对于"相似代码改造"单元，需要新增**兼容性测试**：
  - 测试改造后的代码是否与 kiro 规范兼容
  - 测试改造后的代码是否与现有代码冲突
  - 测试改造后的代码功能是否正常

### 执行流程

```
开始执行步骤 1...
[修改文件]
[如果需要测试] → 测试步骤 1...
[测试结果]
✅ 步骤 1 完成

开始执行步骤 2（相似代码改造）...
[改造文件]
[如果需要测试] → 兼容性测试...
[测试结果：是否与 kiro 规范兼容]
[如果需要测试] → 功能测试...
[测试结果：功能是否正常]
✅ 步骤 2 完成

开始执行步骤 3...
[修改文件]
[如果需要测试] → 测试步骤 3...
[测试结果]
✅ 步骤 3 完成

...

[如果需要测试] → 全流程测试...
[测试结果]
✅ 全流程测试通过

✅ Phase 4 完成

所有功能已实现并测试通过！

接下来进入 Phase 5：总结（如果需要）
```

## 📋 Phase 5：总结（可选）

### 核心动作

与原生开发流程的 Phase 5 相同，但需要额外：
- 在总结文档中记录：
  - 相似代码的来源
  - 改造的关键决策
  - 改造过程中遇到的问题
  - 可复用的改造经验

### 输出物（如果执行）

```
✅ Phase 5 完成

已生成总结文档：
- 文件：requirements/{module}/{version}/summary.md
- 内容：需求概述、实现方案、相似代码改造、关键决策、问题解决、可复用经验

已更新状态：
- current_context.yaml：标记需求为已完成

🎉 二次开发流程全部完成！
```

### 输出物（如果跳过）

```
⏭️ Phase 5 已跳过（用户选择不生成总结文档）

🎉 二次开发流程全部完成！
```

## 📊 流程图

```
Phase 0：评估相似代码
    ↓
    - 分析适配性
    - 评估改造成本
    - 提取可复用片段
    - 确认改造风险
    ↓
    询问用户：继续二次开发 / 改为原生开发 / 取消
    ↓
Phase 1：了解项目与需求
    ↓
    - 确认需求边界
    - 扫查可复用资产（含相似代码）
    - 创建目录结构
    ↓
Phase 2：分析需求
    ↓
    - 拆解为最小单元（含改造单元）
    - 主动澄清不确定点
    - 询问测试和文档需求
    ↓
Phase 3：制定执行计划
    ↓
    - 按"改测"拆分步骤（含改造步骤）
    - 对齐 kiro 规范
    - 根据选择调整计划
    ↓
Phase 4：落地执行
    ↓
    - 执行单个最小单元
    - 改造单元新增兼容性测试
    - 立即测试（如果需要）
    - 测试通过→下一个
    - 测试不通过→回滚修复
    ↓
Phase 5：总结（可选）
    ↓
    - 生成总结文档（含改造经验）
    - 更新状态
    ↓
完成
```

## 🎯 执行约束（强制遵守）

### 约束 1：Phase 0 必须执行
二次开发流程必须先执行 Phase 0 评估，不可跳过。

### 约束 2：评估结果必须明确
Phase 0 的评估结果必须明确给出建议：继续二次开发 / 改为原生开发。

### 约束 3：兼容性测试必须执行
对于"相似代码改造"单元，如果需要测试，必须执行兼容性测试。

### 约束 4：改造原因必须说明
每个改造单元都要明确说明改造原因，不要盲目改造。

### 约束 5：复用原生流程
Phase 1-5 必须复用原生开发流程的逻辑，不要重复定义。
