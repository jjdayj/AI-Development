# VibeCoding 工作流主入口

## 🎯 规则说明

本规则负责：
1. 识别工作流触发关键词
2. 询问用户选择工作流类型
3. 路由到对应的工作流（原生开发 / 二次开发）

**核心理念**：
- 轻量化，适合单人快速开发
- 改一点测一点
- 测试和文档可选
- 保留核心资产（状态管理、目录规范）

---

## 🔔 触发条件

### 原生开发关键词
用户输入中包含以下任一关键词时触发：
- 开发新功能、实现功能、添加功能
- 创建模块、新建模块、设计模块
- 功能设计、架构设计
- 重构、优化架构

### 二次开发关键词
用户输入中包含以下任一关键词时触发：
- 基于 XXX 代码、基于 XXX 功能
- 二次开发、二次改造
- 参考、借鉴、类似
- 改造现有代码

### 排除场景
- 纯粹的问答（如"什么是 VibeCoding？"）
- 查看文件内容（如"查看 README.md"）
- 简单的命令执行（如"运行测试"）
- 简单的文件修改（如"修改配置文件"）

---

## 📋 执行流程

### 步骤 1：识别触发关键词

检查用户输入是否包含触发关键词：
- 如果包含原生开发关键词 → 进入步骤 2
- 如果包含二次开发关键词 → 进入步骤 2
- 如果不包含任何关键词 → 正常作答，不触发工作流

---

### 步骤 2：询问用户选择工作流类型

**强制询问**（必须执行，不可跳过）：

```
🤔 检测到工作流触发关键词

请选择工作流类型：

A. 进入原生功能开发流程
   - 适用场景：从零开始开发新功能
   - 流程：4 阶段（了解需求 → 分析需求 → 制定计划 → 落地执行）
   - 预计时间：2 小时

B. 进入二次开发流程
   - 适用场景：基于现有代码进行改造
   - 流程：5 阶段（评估代码 → 了解需求 → 分析需求 → 制定计划 → 落地执行）
   - 预计时间：3 小时

C. 不进入工作流，正常作答
   - 适用场景：只是简单询问或查看信息

请选择：A / B / C
```

---

### 步骤 3：需求复杂度分析（仅当选择 A 或 B 时执行）

**触发条件**：用户选择了 A（原生功能开发）或 B（二次开发）

**执行动作**：分析用户需求的复杂度，决定是否需要生成需求探索文档

---

#### 3.1 分析需求复杂度

**分析维度**：
1. **需求规模**：
   - 涉及多少个功能点？
   - 涉及多少个文件？
   - 预计代码量是多少？

2. **需求清晰度**：
   - 需求描述是否清晰？
   - 是否有很多不确定的地方？
   - 是否需要多次对话才能理清？

3. **需求独立性**：
   - 是一个独立的需求？
   - 还是包含多个相关的子需求？
   - 子需求之间的依赖关系如何？

4. **技术复杂度**：
   - 技术实现是否复杂？
   - 是否涉及多个技术栈？
   - 是否需要深入研究？

---

#### 3.2 根据分析结果给出建议

**情况 A：需求简单清晰**

**判断标准**：
- 功能点 ≤ 3 个
- 涉及文件 ≤ 5 个
- 需求描述清晰，不确定点 ≤ 2 个
- 是一个独立的需求
- 技术实现相对简单

**输出**：
```
✅ 需求复杂度分析完成

分析结果：
- 需求规模：简单（X 个功能点，Y 个文件）
- 需求清晰度：清晰（Z 个不确定点）
- 需求独立性：独立需求
- 技术复杂度：简单

建议：直接进入工作流，无需生成需求探索文档

接下来进入 [原生功能开发流程 / 二次开发流程]...
```

**执行动作**：
- 直接进入对应的工作流（原生开发 Phase 1 或二次开发 Phase 0）

---

**情况 B：需求较复杂但是单一需求**

**判断标准**：
- 功能点 4-8 个
- 涉及文件 6-15 个
- 需求描述有一定复杂度，不确定点 3-5 个
- 是一个需求，但需要多次对话才能理清
- 技术实现有一定复杂度

**输出**：
```
⚠️ 需求复杂度分析完成

分析结果：
- 需求规模：中等（X 个功能点，Y 个文件）
- 需求清晰度：需要多次对话理清（Z 个不确定点）
- 需求独立性：单一需求
- 技术复杂度：中等

建议：先生成需求探索文档，方便后续对话追踪

需求探索文档将包含：
- 需求概述
- 功能点列表
- 不确定点列表
- 技术方案初步思路
- 后续对话追踪记录

是否生成需求探索文档？
A. 是，生成文档后进入工作流
B. 否，直接进入工作流

请选择：A / B
```

**用户选择处理**：
- **如果用户选择 A**：
  1. 生成需求探索文档：`requirements/exploration/{需求名称}_exploration.md`
  2. 文档内容包括：需求概述、功能点列表、不确定点列表、技术方案、对话追踪区域
  3. 进入对应的工作流

- **如果用户选择 B**：
  1. 直接进入对应的工作流

---

**情况 C：需求复杂且应拆分为多个需求**

**判断标准**：
- 功能点 > 8 个
- 涉及文件 > 15 个
- 需求描述复杂，包含多个相对独立的子需求
- 子需求之间有依赖关系
- 技术实现复杂

**输出**：
```
⚠️ 需求复杂度分析完成

分析结果：
- 需求规模：复杂（X 个功能点，Y 个文件）
- 需求清晰度：包含多个子需求
- 需求独立性：应拆分为 N 个子需求
- 技术复杂度：复杂

建议的需求拆分：
1. 子需求 1：[名称]
   - 功能点：[列出]
   - 依赖：无
   - 优先级：高

2. 子需求 2：[名称]
   - 功能点：[列出]
   - 依赖：子需求 1
   - 优先级：中

3. 子需求 3：[名称]
   - 功能点：[列出]
   - 依赖：子需求 1, 2
   - 优先级：低

建议：先生成需求探索文档，记录需求拆分和依赖关系

需求探索文档将包含：
- 总体需求概述
- 子需求列表（含依赖关系）
- 开发顺序建议
- 技术方案
- 后续对话追踪记录

是否生成需求探索文档？
A. 是，生成文档后选择一个子需求开始
B. 否，直接选择一个子需求开始

请选择：A / B
```

**用户选择处理**：
- **如果用户选择 A**：
  1. 生成需求探索文档：`requirements/exploration/{总需求名称}_exploration.md`
  2. 文档内容包括：总体概述、子需求列表、依赖关系图、开发顺序、技术方案、对话追踪区域
  3. 询问用户选择哪个子需求开始
  4. 进入对应的工作流（针对选中的子需求）

- **如果用户选择 B**：
  1. 询问用户选择哪个子需求开始
  2. 直接进入对应的工作流（针对选中的子需求）

---

**情况 D：需求过于复杂，不适合 VibeCoding 工作流**

**判断标准**：
- 功能点 > 15 个
- 涉及文件 > 30 个
- 需求描述极其复杂，包含大量不确定点
- 即使拆分后每个子需求仍然很复杂
- 技术实现极其复杂，需要深入研究

**输出**：
```
❌ 需求复杂度分析完成

分析结果：
- 需求规模：极其复杂（X 个功能点，Y 个文件）
- 需求清晰度：极其复杂，大量不确定点
- 需求独立性：即使拆分后仍然很复杂
- 技术复杂度：极其复杂

建议：此需求不适合 VibeCoding 轻量化工作流

原因：
- VibeCoding 工作流适合单人快速开发，预计时间 2-3 小时
- 此需求预计需要 X 小时，远超工作流设计范围
- 建议使用更完整的项目管理流程

替代方案：
1. 将需求进一步拆分为更小的子需求，每个子需求单独使用 VibeCoding 工作流
2. 使用传统的项目管理方式（需求分析 → 设计 → 开发 → 测试）
3. 寻求团队协作，而不是单人开发

是否继续？
A. 继续拆分需求，使用 VibeCoding 工作流
B. 退出工作流，正常作答

请选择：A / B
```

**用户选择处理**：
- **如果用户选择 A**：
  1. 引导用户进一步拆分需求
  2. 重新执行需求复杂度分析
  3. 如果拆分后符合标准，进入对应的工作流

- **如果用户选择 B**：
  1. 退出工作流
  2. 正常作答

---

### 步骤 4：路由到对应工作流

**前置条件**：已完成需求复杂度分析，确认可以进入工作流

---

#### 如果用户选择 A（原生功能开发）

```
✅ 开始执行原生功能开发流程

接下来将执行 4 个阶段：
- Phase 1：了解项目与需求（快速对齐）
- Phase 2：分析需求（主动澄清）
- Phase 3：制定执行计划（改测循环）
- Phase 4：落地执行（小步快跑）
- Phase 5：总结（可选）

预计时间：2 小时

开始执行 Phase 1：了解项目与需求...
```

**执行动作**：
- 加载并执行 `native_development.md` 中的原生开发流程
- 从 Phase 1 开始

---

#### 如果用户选择 B（二次开发）

```
✅ 开始执行二次开发流程

接下来将执行 5 个阶段：
- Phase 0：评估相似代码（适配性分析）
- Phase 1：了解项目与需求（快速对齐）
- Phase 2：分析需求（主动澄清）
- Phase 3：制定执行计划（改测循环）
- Phase 4：落地执行（小步快跑）
- Phase 5：总结（可选）

预计时间：3 小时

开始执行 Phase 0：评估相似代码...
```

**执行动作**：
- 加载并执行 `secondary_development.md` 中的二次开发流程
- 从 Phase 0 开始

---

#### 如果用户选择 C（不进入工作流）

```
✅ 已选择：不进入工作流

我将正常回答你的问题，不执行工作流程。

如果后续需要进入工作流，请明确说明：
- "进入原生功能开发流程"
- "进入二次开发流程"
```

**执行动作**：
- 正常作答，不触发任何工作流
- 不创建目录结构
- 不生成 meta.yaml

## 🎯 执行约束（强制遵守）

### 约束 1：必须询问用户选择工作流类型
检测到触发关键词后，**必须**询问用户选择工作流类型，不可自动判断。

### 约束 2：询问格式固定且使用中文
询问格式必须严格按照上述模板，使用中文，包含：
- 三个选项（A/B/C）
- 每个选项的适用场景
- 每个选项的流程说明
- 每个选项的预计时间
- 明确的选择提示："请选择：A / B / C"

### 约束 3：选择 A 或 B 后必须执行需求复杂度分析
如果用户选择 A 或 B，**必须**先执行需求复杂度分析，不可跳过。

### 约束 4：需求复杂度分析使用中文
需求复杂度分析的所有输出和询问都必须使用中文。

### 约束 5：根据复杂度分析结果给出明确建议
需求复杂度分析必须给出明确的建议：
- 情况 A：直接进入工作流
- 情况 B：建议生成探索文档（询问用户）
- 情况 C：建议生成探索文档并拆分需求（询问用户）
- 情况 D：通知用户不适合此工作流（询问用户是否继续）

### 约束 6：生成需求探索文档的格式
如果需要生成需求探索文档，必须包含：
- 需求概述
- 功能点列表
- 不确定点列表
- 技术方案初步思路
- 后续对话追踪区域（空白，供后续填写）

### 约束 7：选择 C 后不触发任何工作流
如果用户选择 C，则完全不触发工作流，不执行需求复杂度分析，直接正常作答。

### 约束 8：用户选择后立即执行
用户在每个询问环节做出选择后，立即执行对应的动作，不再重复询问。
