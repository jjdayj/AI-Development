## 一、架构核心风险修正（必须修正，影响三模块联动落地）

### 1.1 跨模块依赖顺序未做强制校验

**问题**：三个模块有严格的依赖关系（动态模块加载器 → Git 提交自动化 → VibCoding 工作流增强），但原计划均未在关键节点设置前置依赖可用性校验点，会导致提前开发下游模块时因上游基座未完成、格式不兼容出现大量返工。

**修正方案**：

1. **Git 提交自动化模块**（tasks-v2.md）：
   - 新增阶段 0：前置校验与环境初始化
   - 任务 0.1：验证动态模块加载器已完成核心实现
   - 任务 0.2：验证 current_context.yaml 格式规范已冻结
   - 任务 0.3：验证模块加载机制可用
   - 任务 0.4：创建上游依赖兼容性校验工具

2. **VibCoding 工作流增强模块**（tasks-v2.md）：
   - 新增阶段 0：前置校验与环境初始化
   - 任务 0.1：验证动态模块加载器已完成核心实现
   - 任务 0.2：验证 Git 提交模块已完成核心实现
   - 任务 0.3：验证数据格式完全兼容
   - 任务 0.4：创建端到端兼容性校验工具

3. **所有模块的 Checkpoint**：
   - 每个 Checkpoint 必须包含「上游依赖兼容性校验」
   - 确保当前模块的输出符合上游/下游的格式要求

**影响**：确保模块开发顺序正确，避免因格式不兼容导致的返工。

---

### 1.2 核心属性测试被标记为「可选」

**问题**：三份计划均把属性测试标记为可选（*），但属性测试是验证设计文档中「正确性属性」的唯一手段，是保障核心逻辑不出错的核心抓手。标记为可选会导致核心逻辑（格式校验、round-trip 一致性、逻辑组合正确性）无强验证，后期出现隐性 bug 的风险极高。

**修正方案**：

1. **动态模块加载器**（tasks-v2.md）：
   - 属性 1-15（所有核心正确性属性）改为必选
   - 仅边缘场景测试（如 Workflow 版本独立性）保留可选标记
   - 每个属性测试最少 100 次迭代

2. **Git 提交自动化模块**（tasks-v2.md）：
   - 属性 1-28（所有核心正确性属性）改为必选
   - 仅 CLI 相关测试保留可选标记
   - 每个属性测试最少 100 次迭代

3. **VibCoding 工作流增强模块**（tasks-v2.md）：
   - 由于是 Prompt 工程，属性测试转化为标准化手动测试用例
   - 所有核心功能的测试用例均为必选
   - 创建 test-cases.md 文档，包含测试场景、预期结果和执行步骤

**影响**：确保核心逻辑的正确性，避免隐性 bug。
