# VibeCoding 工作流项目

## 项目简介

这是一个基于 Kiro AI 的轻量化开发工作流系统，专为单人快速开发设计。采用"改一点测一点"的 VibeCoding 理念，支持原生功能开发和二次开发两种场景，帮助开发者高效完成 2-3 小时内的快速迭代开发。

## 🚀 核心特性

### 🎯 VibeCoding 工作流系统
- **轻量化流程**：原生开发 4 阶段，二次开发 5 阶段
- **需求复杂度分析**：自动分析需求，避免过于复杂的需求
- **改一点测一点**：小步快跑，每个最小单元完成后立即测试
- **可选测试和文档**：根据需求灵活配置，不强制执行
- **支持二次开发**：评估现有代码，智能改造和复用
- **中文交互**：所有询问和分析都使用中文

## ⚡ 快速开始

### 1. 环境准备

确保你已经安装了 Kiro AI。

### 2. 使用工作流

#### 触发工作流

在 Kiro 中输入触发关键词：
```
开发新功能
```

或者：
```
实现功能
创建模块
功能设计
重构
```

#### 选择工作流类型

系统会询问你选择（中文）：
```
🤔 检测到工作流触发关键词

请选择工作流类型：
A. 进入原生功能开发流程（4 阶段，2 小时）
B. 进入二次开发流程（5 阶段，3 小时）
C. 不进入工作流，正常作答

请选择：A / B / C
```

#### 需求复杂度分析

如果选择 A 或 B，系统会分析需求复杂度（中文）：
- **简单需求**：直接进入工作流
- **中等需求**：建议生成需求探索文档
- **复杂需求**：建议拆分为多个子需求
- **过于复杂**：通知用户不适合此工作流

## 🔄 工作流程

### 完整流程图

```
用户输入触发关键词
    ↓
询问选择工作流类型（中文）
    ├─ A. 原生功能开发
    ├─ B. 二次开发
    └─ C. 不进入工作流
    ↓
需求复杂度分析（仅 A/B，中文）
    ├─ 简单：直接进入工作流
    ├─ 中等：建议生成探索文档
    ├─ 复杂：建议拆分需求
    └─ 过于复杂：通知用户
    ↓
进入对应工作流
```

### 原生功能开发流程（4 阶段）

1. **Phase 1：了解项目与需求**
   - 确认需求边界
   - 快速扫查可复用资产
   - 创建目录结构（如果需要）

2. **Phase 2：分析需求**
   - 拆解为最小可执行单元
   - 主动澄清所有不确定点
   - 询问：测试需求、文档需求、GUI 偏好

3. **Phase 3：制定执行计划**
   - 按"改一点测一点"拆分步骤
   - 对齐项目规范

4. **Phase 4：落地执行**
   - 按计划执行最小单元开发
   - 每个单元完成后立即测试
   - 测试通过→下一个单元；不通过→回滚修复

5. **Phase 5：总结**（可选）
   - 根据 Phase 2 的选择决定是否生成文档

### 二次开发流程（5 阶段）

1. **Phase 0：评估相似代码**
   - 分析适配性（兼容性、改造成本）
   - 提取可复用片段
   - 确认改造风险

2. **Phase 1-4**：复用原生开发流程
   - Phase 1：将相似代码复用计划加入资产清单
   - Phase 3：将代码改造拆分为最小单元
   - Phase 4：新增兼容性测试

3. **Phase 5：总结**（可选）

## ⚙️ 配置说明

### 全局配置

编辑 `kiro/config.yaml`：

```yaml
modules:
  workflow:
    enabled: true   # 启用/禁用工作流
    version: v2.0   # 工作流版本
    priority: 200   # 优先级
```

### 工作流配置

编辑 `kiro/modules/workflow/v2.0/config.yaml`：

```yaml
# 默认行为配置
defaults:
  testing:
    enabled: false  # 默认关闭测试
    ask_user: true  # 每次询问用户
  
  documentation:
    enabled: true   # 默认生成文档
    ask_user: true  # 每次询问用户

# 需求复杂度分析标准
complexity_analysis:
  simple:
    function_points: 3      # 功能点数量
    files: 5                # 涉及文件数量
  
  medium:
    function_points: 8
    files: 15
  
  complex:
    function_points: 15
    files: 30
```

## 🚨 常见问题

### Q: 如何禁用工作流？
A: 在 `kiro/config.yaml` 中将 `workflow.enabled` 改为 `false`

### Q: 如何修改复杂度分析标准？
A: 编辑 `kiro/modules/workflow/v2.0/config.yaml` 中的 `complexity_analysis` 部分

### Q: 如何修改默认的测试和文档行为？
A: 编辑 `kiro/modules/workflow/v2.0/config.yaml` 中的 `defaults` 部分

### Q: 工作流适合什么样的项目？
A: 适合单人开发、快速迭代、2-3 小时内可完成的功能开发

### Q: 什么样的需求不适合这个工作流？
A: 功能点 > 15 个、涉及文件 > 30 个、需要深入研究的复杂需求
