## 🎯 核心架构决策（ADR）

### ADR-001：采用"模块优先"目录结构
- **理由**：支持模块独立发布、独立授权、独立文档
- **结构**：`.kiro/modules/{module}/{version}/{steering,scripts,configs}/`

### ADR-002：不使用软链接，改用配置文件管理版本
- **理由**：跨平台兼容性、Git 友好、更安全
- **方案**：使用 `kiro.yaml` 指定当前激活的模块版本

### ADR-003：doc 目录物理分离
- **理由**：清晰区分个人日志和项目文档
- **结构**：`doc/personal/`（排除）+ `doc/project/`（纳入 Git）

### ADR-004：changelog 增加"效果评估"字段
- **理由**：数据驱动的提示词迭代优化
- **字段**：修改内容、变更原因、效果评估、测试用例、关联文件、回滚风险

## 🔄 核心开发流程

本项目采用 **VibCoding（振动式编码）** 作为核心开发方法论。

### 什么是 VibCoding？

VibCoding 是一种强调**快速反馈、小步迭代、持续调整**的开发思维，包含 4 个阶段 15 个步骤：

1. **Phase 1: 探索与决策**（需求探索 → 架构决策 → 需求细化）
2. **Phase 2: 快速原型**（MVP 实现 → 快速测试 → 反馈收集）
3. **Phase 3: 完善与加固**（测试计划 → 基线测试 → 完整实现 → 完整测试 → 文档更新）
4. **Phase 4: 审查与发布**（代码审查 → 发布准备 → 部署上线 → 监控反馈）

**详细流程**：参见 `16-VibCoding开发流程指南.md`

### 本项目的应用

- **阶段 0-2**（已完成）：使用了 Phase 1（探索与决策）
- **阶段 3**（待执行）：推荐使用 Phase 2（快速原型）+ Phase 3（完善加固）
  - 迭代 1：只迁移 ai_dev.md → 测试 → 反馈（10分钟）
  - 迭代 2：迁移 ai-dev-script.py → 测试 → 反馈（10分钟）
  - 迭代 3：迁移 ai-dev-config.json → 测试 → 反馈（10分钟）
  - Phase 3：完善与加固 → 完整测试 → 文档更新（20分钟）
- **阶段 4-5**（未来）：使用完整的 VibCoding 循环

**核心优势**：降低风险、快速反馈、小步迭代、容易调整
