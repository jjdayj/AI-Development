# Git 回滚功能需求探索

## 需求来源
2026-03-02 用户提出新需求：需要添加 Git 回滚的 prompt 需求，如果某次对话错误后，能够按照指定标准进行回滚。

## 需求背景

### 问题场景
1. 某次对话中，Kiro 做了错误的修改
2. 修改已经提交到 Git
3. 用户希望能够快速回滚到之前的正确状态

### 当前痛点
- 没有标准的回滚流程
- 不清楚如何安全地回滚
- 可能需要手动操作 Git 命令
- 担心回滚操作会造成更多问题

## 需求探索（5个核心疑问）

### 疑问 1：回滚的触发条件是什么？
**可能的场景**：
- A. 用户发现某次对话的修改有问题
- B. 测试失败，需要回滚到上一个稳定版本
- C. 误操作，需要撤销最近的提交
- D. 需要回到某个特定的历史版本

**需要明确**：
- 什么情况下需要回滚？
- 回滚的粒度是什么？（单次提交 vs 多次提交）
- 是否需要保留回滚记录？

### 疑问 2：回滚的目标是什么？
**可能的目标**：
- A. 回滚到上一次提交
- B. 回滚到指定的 commit ID
- C. 回滚到某个标签（tag）
- D. 回滚到某个时间点

**需要明确**：
- 如何指定回滚目标？
- 是否需要预览回滚内容？
- 回滚后是否需要验证？

### 疑问 3：回滚的方式是什么？
**Git 回滚方式对比**：

| 方式 | 命令 | 特点 | 适用场景 |
|------|------|------|----------|
| **git reset --soft** | `git reset --soft HEAD~1` | 保留修改在暂存区 | 想重新提交 |
| **git reset --mixed** | `git reset --mixed HEAD~1` | 保留修改在工作区 | 想重新修改 |
| **git reset --hard** | `git reset --hard HEAD~1` | 完全删除修改 | 确定要丢弃 |
| **git revert** | `git revert <commit>` | 创建新提交撤销 | 已推送到远程 |

**需要明确**：
- 使用哪种回滚方式？
- 是否需要强制推送？
- 如何处理远程仓库？

### 疑问 4：回滚的安全性如何保证？
**安全考虑**：
- A. 回滚前是否需要备份？
- B. 回滚前是否需要确认？
- C. 回滚后如何验证？
- D. 回滚失败如何恢复？

**需要明确**：
- 是否需要创建备份分支？
- 是否需要用户确认？
- 回滚后如何测试？

### 疑问 5：回滚的流程是什么？
**可能的流程**：
1. 用户发现问题 → 请求回滚
2. Kiro 分析提交历史 → 确定回滚目标
3. Kiro 执行回滚操作 → 验证结果
4. Kiro 更新文档 → 记录回滚原因

**需要明确**：
- 流程的每个步骤是什么？
- 是否需要自动化？
- 是否需要人工介入？

## 初步需求分析

### 核心需求
1. **回滚触发**：用户明确指出某次对话有问题，需要回滚
2. **回滚目标**：能够指定回滚到哪个提交（通过 commit ID 或描述）
3. **回滚方式**：根据情况选择合适的回滚方式
4. **安全保障**：回滚前确认，回滚后验证
5. **文档记录**：记录回滚原因和过程

### 边界情况
1. 回滚到的提交不存在
2. 回滚会影响其他人的工作（多人协作）
3. 回滚后发现还是有问题（需要再次回滚）
4. 回滚操作本身失败

### 用户体验
- 用户只需要说"回滚到上一次提交"或"回滚到 xxx 提交"
- Kiro 自动分析并执行回滚
- Kiro 提供清晰的回滚报告
- 用户可以验证回滚结果

## 补充需求探索：Git 回滚功能全局开关

### 需求来源
2026-03-02 用户补充：需要一个独立的开关，控制是否允许通过 prompt 进行 Git 回滚。

### 需求明确

#### 功能描述
在 `kiro.yaml` 中添加一个全局开关，控制 Git 回滚功能是否启用。

**开关位置**：
```yaml
# 实验性功能开关（可选）
experimental:
  # 是否启用自动 RFC 记录
  auto_rfc_recording: false
  
  # 是否启用提示词效果评估
  prompt_effect_evaluation: true
  
  # 是否启用 Git 自动回滚功能（新增）
  git_auto_rollback: false
```

#### 行为定义

**当 `git_auto_rollback: true` 时**：
- ✅ 用户可以通过 prompt 触发 Git 回滚
- ✅ Kiro 会解析回滚指令并执行
- ✅ 正常的回滚流程（确认 → 备份 → 回滚 → 验证）

**当 `git_auto_rollback: false` 时**：
- ❌ 用户输入回滚指令时，Kiro 提示：
  ```
  ⚠️ Git 自动回滚功能已关闭
  
  如需启用，请修改 .kiro/kiro.yaml：
  experimental:
    git_auto_rollback: true
  
  或者手动执行 Git 命令：
  git revert HEAD
  ```
- ❌ 不执行任何回滚操作
- ✅ 用户仍可以手动执行 Git 命令

#### 使用场景

**为什么需要这个开关？**
1. **安全性**：防止误操作导致的意外回滚
2. **权限控制**：团队协作时，可能只允许特定人员回滚
3. **学习阶段**：新用户可能不熟悉回滚，先关闭功能
4. **生产环境**：生产环境可能禁止自动回滚

**什么时候开启？**
- 个人开发环境
- 熟悉回滚流程后
- 需要快速修复错误时

**什么时候关闭？**
- 团队协作环境
- 生产环境
- 学习阶段
- 不确定是否需要回滚时

#### 配置示例

**开发环境（推荐开启）**：
```yaml
experimental:
  git_auto_rollback: true  # 允许快速回滚
```

**生产环境（推荐关闭）**：
```yaml
experimental:
  git_auto_rollback: false  # 禁止自动回滚
```

**团队协作（根据团队规范）**：
```yaml
experimental:
  git_auto_rollback: false  # 统一由项目负责人回滚
```

### 需求确认

✅ 需求已明确：
- 在 `kiro.yaml` 的 `experimental` 部分添加 `git_auto_rollback` 开关
- 默认值：`false`（安全优先）
- 关闭时提示用户如何启用
- 不影响手动 Git 命令
