### 第 1 部分：规则说明

```markdown
# VibCoding 开发流程规则

## 🎯 规则说明

本规则定义了 VibCoding（振动式编码）开发流程的强制执行规范。

**核心目标**：
- 防止需求理解错误（如 Git 提交工作流案例）
- 确保目标用户和使用场景在需求探索阶段就明确
- 遇到不确定信息时主动提问，而不是自行假设

**适用范围**：
- 所有新功能开发
- 涉及架构调整的 Bug 修复
- 模块化重构

**不适用范围**：
- 简单的配置修改
- 纯文档编写
- 简单的 Bug 修复（不涉及架构）
```

```markdown
### 步骤 1.1：目标用户识别（强制）

**提问方式**：
```

A. Agent（AI 助手内部调用）
   - Agent 在对话中识别关键词，自动执行操作
   - 用户不需要手动运行任何脚本或命令
   
B. 人类用户（手动运行脚本/命令）
   - 用户需要手动运行 Python 脚本或命令行工具
   - 功能作为独立工具存在
   
C. 两者都是
   - 既支持 Agent 自动调用，也支持用户手动运行

请选择：A / B / C

如果不确定，请说明你的使用场景，我来帮你判断。

1. 用户是否需要手动运行脚本？

   - 如果否 → 选择 A（Agent）

2. 用户是否在对话中说一句话就能触发？

   - 如果否 → 选择 B（人类用户）

```markdown
## ❌ 错误案例参考

### 案例 1：Git 提交工作流（错误版本）

**需求**：
用户："我想开发一个 Git 提交工作流功能，用户说'提交所有修改'就能自动提交"

**错误的理解**：
- 目标用户：人类用户（❌ 错误）
- 使用场景：独立工具（❌ 错误）
- 触发方式：命令行参数 `python git_commit.py "提交所有修改"`（❌ 错误）
- 交互方式：命令行交互（❌ 错误）

**错误的实现**：
- 创建了独立的 Python 脚本 `git_commit.py`
- 用户需要手动运行脚本
- 完全偏离了原始需求

**正确的理解**：
- 目标用户：Agent（✅ 正确）
- 使用场景：Agent 内部调用（✅ 正确）
- 触发方式：Steering 规则识别关键词（✅ 正确）
- 交互方式：Agent 与用户对话（✅ 正确）
- 集成方式：Steering 规则（✅ 正确）

**正确的实现**：
- 创建 `.kiro/steering/git_commit.md`
- Agent 识别关键词"提交"
- Agent 执行 git 命令
- Agent 与用户对话确认

**教训**：
- 必须在需求探索阶段就明确目标用户
- "用户说'提交所有修改'" ≠ "用户运行 `python git_commit.py '提交所有修改'`"
- "用户说" = Agent 识别关键词

---

### 案例 2：AI Development（正确版本）

**需求**：
用户："我想开发一个项目管理工具，通过中文指令创建项目目录"

**正确的理解**：
- 目标用户：人类用户（✅ 正确）
- 使用场景：独立工具（✅ 正确）
- 触发方式：命令行参数（✅ 正确）
- 交互方式：命令行交互（✅ 正确）

**正确的实现**：
- 创建了 Python 脚本 `ai-dev-script.py`
- 创建了 Steering 规则 `ai-dev.md`（用于生成命令）
- 用户手动运行脚本
- 符合原始需求

**为什么这个是正确的**：
- 用户明确说了"工具"，说明是独立工具
- 用户需要"创建项目目录"，说明是人类用户手动操作
- 不是 Agent 自动执行的功能
```

```markdown
## 📚 规则总结

### 核心原则

   - 如果错误，立即回到需求探索阶段

| 维度 | Agent 内部调用 | 独立工具 |

   - 原因：没有明确使用场景是独立工具
   - 解决：在需求探索阶段强制提问
```
