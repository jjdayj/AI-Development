# 工作流补充：问题总结与反思机制

**文档创建时间**: 2026-03-02  
**文档版本**: v1.0  
**目标**: 在工作流中增加问题总结和反思机制，避免重复犯错

---

## 🎯 核心问题

当前的双轨工作流（Prompt 工作流 + 代码工作流）缺少了一个关键环节：

❌ **没有要求在遇到问题时记录总结和反思**

这导致：
- 同样的错误可能重复发生（如 Git 提交工作流的错误）
- 经验教训没有被系统化记录
- 无法形成知识积累

---

## 📋 解决方案：增加 Phase 5

### 原有流程
```
Phase 1: 探索与决策
Phase 2: 快速原型 / 规则设计与实现
Phase 3: 完善与加固 / 轻量级验证
Phase 4: 审查与发布
```

### 新增流程
```
Phase 1: 探索与决策
Phase 2: 快速原型 / 规则设计与实现
Phase 3: 完善与加固 / 轻量级验证
Phase 4: 审查与发布
Phase 5: 总结与反思（新增）⭐
```

---

## 📝 Phase 5: 总结与反思（强制执行）

### 触发条件

#### 条件 1：项目完成时（必须执行）
无论项目成功还是失败，都必须执行 Phase 5。

#### 条件 2：遇到重大问题时（可选执行）
在 Phase 2-4 过程中，如果遇到以下情况，可以提前执行 Phase 5：
- 需求理解错误（如 Git 提交工作流案例）
- 架构设计错误
- 实现方向错误
- 需要回滚重做

---

### 执行内容

#### 🔴 强制项 1：问题识别与分类

**问题分类**：
```
A. 需求理解问题
   - 目标用户理解错误
   - 使用场景理解错误
   - 触发方式理解错误
   - 交互方式理解错误

B. 架构设计问题
   - 缺少 Agent 集成层
   - 模块划分不合理
   - 技术选型错误

C. 实现问题
   - 代码质量问题
   - 测试覆盖不足
   - 性能问题

D. 流程问题
   - 跳过了必要步骤
   - 没有充分验证
   - 文档不完整

E. 其他问题
   - 请说明
```

**记录方式**：
```
📋 问题识别

问题类型：[A/B/C/D/E]
问题描述：[详细描述问题是什么]
发现时间：[Phase 几发现的]
影响范围：[影响了哪些部分]
```

#### 🔴 强制项 2：根本原因分析

**分析框架（5 Why 分析法）**：
```
问题：[问题描述]

Why 1：为什么会发生这个问题？
→ [第一层原因]

Why 2：为什么会有第一层原因？
→ [第二层原因]

Why 3：为什么会有第二层原因？
→ [第三层原因]

Why 4：为什么会有第三层原因？
→ [第四层原因]

Why 5：为什么会有第四层原因？
→ [根本原因]

根本原因总结：
[用一句话总结根本原因]
```

**示例（Git 提交工作流）**：
```
问题：将 Agent 内部调用的功能开发成了独立工具

Why 1：为什么会开发成独立工具？
→ 因为在架构设计阶段就设计成了独立工具架构

Why 2：为什么会设计成独立工具架构？
→ 因为在需求探索阶段就理解成了独立工具

Why 3：为什么会理解成独立工具？
→ 因为需求探索阶段没有明确目标用户是 Agent

Why 4：为什么没有明确目标用户？
→ 因为 VibCoding 流程没有强制要求明确目标用户

Why 5：为什么 VibCoding 流程没有强制要求？
→ 因为 VibCoding 流程缺少强制检查点

根本原因总结：
VibCoding 流程缺少强制检查点，导致需求探索阶段没有明确目标用户
```

**教训分类**：
```

   - 不应该做什么？
   - 如何避免类似错误？

B. 架构设计教训
   - 应该做什么？
   - 不应该做什么？
   - 如何避免类似错误？

C. 实现教训
   - 应该做什么？
   - 不应该做什么？
   - 如何避免类似错误？

D. 流程教训
   - 应该做什么？
   - 不应该做什么？
   - 如何避免类似错误？
```

**记录方式**：
```
💡 经验教训

教训 1：[教训标题]
- 应该做：[具体做法]
- 不应该做：[具体做法]
- 如何避免：[预防措施]

教训 2：[教训标题]
- 应该做：[具体做法]
- 不应该做：[具体做法]
- 如何避免：[预防措施]

...
```

**示例（Git 提交工作流）**：
```
💡 经验教训

教训 1：需求探索阶段必须明确目标用户
- 应该做：在 Phase 1 强制回答 5 个问题（目标用户、使用场景、触发方式、交互方式、集成方式）
- 不应该做：跳过目标用户识别，直接进入架构设计
- 如何避免：在 VibCoding 流程中增加强制检查点

教训 2：架构设计必须包含 Agent 集成层
- 应该做：如果目标用户是 Agent，必须设计 Agent 集成层
- 不应该做：设计成独立工具架构
- 如何避免：在架构决策阶段增加 Agent 集成层检查清单

教训 3：参考现有功能时要理解本质
- 应该做：理解 Git 回滚功能的实现方式（Steering 规则 + git 命令）
- 不应该做：只看表面，误以为需要复杂的 Python 模块
- 如何避免：在需求探索阶段参考类似功能的实现方式
```
