## 👋 欢迎使用双轨工作流

这是 Prompt 基座项目的一个核心功能模块，让 AI 助手 Kiro 能够根据你的需求，自动选择最合适的工作方式。

**一句话总结**：写规则用轻量级流程，写代码用完整流程，简单修改直接跳过。

---

## 🎯 这个功能能帮你做什么？

### 问题 1：不同类型的工作需要不同的流程

**以前的困扰**：
- 改一行配置也要走 3 小时的完整流程？太浪费时间！
- 开发复杂功能却用简化流程？质量没保障！

**现在的解决方案**：
- ⚡ 简单修改（改配置、修错别字）→ 快速模式（5-10 分钟）
- 📝 写规则/文档 → Prompt 工作流（2-3 小时，轻量级）
- 💻 写代码/开发功能 → 代码工作流（5-8 小时，完整版）

---

### 问题 2：每次都要手动选择工作流？

**以前的困扰**：
- 每次开始工作都要手动配置
- 同一个项目只能用一种工作流
- 容易选错，导致流程不匹配

**现在的解决方案**：
- ✅ 自动识别（根据文件类型）
- ✅ 智能切换（同一项目可以同时用两种流程）
- ✅ 支持手动覆盖（你说了算）

---

### 问题 3：项目做完就忘了，下次还犯同样的错？

**以前的困扰**：
- 遇到问题解决了，但没记录
- 下次遇到同样的问题，又要重新摸索
- 经验没有积累，团队无法共享

**现在的解决方案**：
- ✅ 每个项目结束后强制总结反思
- ✅ 自动生成总结文档
- ✅ 记录问题、原因、教训、改进措施
- ✅ 形成知识库，避免重复犯错

---

## 🚀 核心特性（5 个）

### 特性 1：智能识别工作流 ⭐⭐⭐⭐⭐

**旧版（不够灵活）**：
```yaml
project:
  type: "prompt-base"  # 整个项目只能选一种
```

**新版（智能切换）**：
```yaml
workflow:
  default: "prompt"
  auto_switch:
    - pattern: ".kiro/steering/*.md"
      workflow: "prompt"
    - pattern: "**/*.py"
      workflow: "code"
```

**实际效果**：
```
你说："修改 .kiro/steering/git_rollback.md"
→ 自动识别：Steering 规则文件
→ 自动选择：Prompt 工作流（轻量级）

你说："开发 scripts/data_processor.py"
→ 自动识别：Python 代码
→ 自动选择：代码工作流（完整版）
```

**优势**：
- ✅ 不用手动切换
- ✅ 按文件类型自动识别
- ✅ 同一项目可以同时开发规则和代码

---

### 特性 2：快速跳过机制 ⭐⭐⭐⭐⭐

**快速跳过配置**：
```yaml
quick_skip:
  enabled: true
  
  # 手动跳过关键词
  manual_skip_keywords:
    - "快速修改"
    - "简单修改"
    - "跳过流程"
  
  # 自动跳过条件
  auto_skip_conditions:
    - type: "line_count"
      threshold: 10
      description: "修改行数 < 10 行"
    
    - type: "file_type"
      patterns: ["*.yaml", "*.json", "*.txt"]
      description: "简单配置文件"
    
    - type: "change_type"
      types: ["typo", "format", "comment"]
      description: "错别字、格式化、注释修改"
```

**使用示例**：
```
你说："快速修改 kiro.yaml，添加一个配置项"

Kiro：
🤔 检测到快速修改关键词

这是一个简单修改吗？可以跳过完整的工作流吗？

适合跳过的情况：
- 简单配置修改（< 10 行）
- 纯文档错别字修正
- 简单的规则微调
- 格式化、注释修改

请确认：
A. 是，跳过流程（5-10 分钟）
B. 否，执行完整流程（2.5-8 小时）

你说："A"

Kiro：
✅ 已跳过完整工作流
直接执行修改...
```

**优势**：
- ✅ 简单修改不走完整流程
- ✅ 节省时间（从 2-3 小时降到 5-10 分钟）
- ✅ 用户确认机制（避免误跳过）

---

### 特性 3：两套工作流对比 ⭐⭐⭐⭐

**Prompt 工作流（轻量级）**：
- 适合：Steering 规则、Hook 规则、文档
- 时间：2.5-3.5 小时
- 测试：手动测试即可
- 特点：快速、简洁

**代码工作流（完整版）**：
- 适合：Python 代码、复杂功能
- 时间：5-8 小时
- 测试：单元测试 + 集成测试（覆盖率 > 80%）
- 特点：严谨、完整

**对比表格**：

| 维度 | Prompt 工作流 | 代码工作流 |
|------|--------------|-----------|
| **适用场景** | 规则、文档 | 代码、功能 |
| **时间投入** | 2.5-3.5 小时 | 5-8 小时 |
| **测试要求** | 手动测试 | 自动化测试 |
| **复杂度** | 轻量级 | 完整版 |

---

### 特性 4：强制总结反思 ⭐⭐⭐⭐⭐

**问题**：项目做完就忘了，下次还犯同样的错误。

**解决方案**：每个项目结束后，强制执行总结反思（Phase 5）

**总结内容（5 个强制项）**：
1. 🔴 遇到了哪些问题？
2. 🔴 根本原因是什么？（5 Why 分析法）
3. 🔴 学到了什么经验教训？
4. 🔴 下次怎么改进？
5. 🔴 知识如何沉淀？

**输出文档**：
```
doc/project/
└── YYYYMMDD_项目名-总结与反思.md
    - 项目背景和结果
    - 问题识别与分类
    - 根本原因分析
    - 经验教训
    - 改进措施
    - 知识沉淀
```

**优势**：
- ✅ 避免重复犯错
- ✅ 形成知识积累
- ✅ 团队经验共享

---

### 特性 5：防止需求理解错误 ⭐⭐⭐⭐⭐

**问题**：开发了半天，发现理解错了需求，白干了。

**解决方案**：Phase 1 的 5 个强制检查项

**5 个必答问题**：
1. 🔴 目标用户是谁？（Agent / 人类用户 / 第三方应用）
2. 🔴 使用场景是什么？（Agent 内部调用 / 独立工具 / API 服务）
3. 🔴 如何触发？（Steering 规则 / 命令行 / API）
4. 🔴 如何交互？（Agent 对话 / 命令行交互 / 无交互）
5. 🔴 如何集成？（如果目标用户包含 Agent）

**关键规则**：
- Agent 遇到不确定信息时，必须主动提问
- 不得自行假设
- 必须得到明确答案才能继续

**案例对比**：

**❌ 错误案例：Git 提交工作流（已回滚）**
```
错误理解：将 Agent 内部调用理解成独立工具
根本原因：Phase 1 没有明确目标用户
结果：开发了独立 Python 脚本，但实际需要 Steering 规则
教训：必须在需求探索阶段强制回答 5 个问题
```

**✅ 正确案例：Git 回滚功能**
```
目标用户：Agent
使用场景：Agent 内部调用
触发方式：Steering 规则识别关键词
实现方式：Steering 规则 + git 命令
工作流：Prompt 工作流
结果：成功实现，符合预期
```

## 🎯 常见问题

### Q1：什么时候用 Prompt 工作流？
**A**：开发 Steering 规则、Hook 规则、文档时使用。特点是轻量级，不需要写自动化测试。

### Q2：什么时候用代码工作流？
**A**：开发 Python 代码、复杂功能时使用。特点是完整版，需要写单元测试和集成测试。

### Q3：什么时候可以快速跳过？
**A**：简单修改时可以跳过，比如：
- 修改配置文件的一个参数
- 修正文档的错别字
- 简单的格式化调整
- 修改注释

### Q4：Phase 5 可以跳过吗？
**A**：不可以。Phase 5 是强制执行的，用于总结经验教训，避免重复犯错。

### Q5：如何手动指定工作流？
**A**：在需求中明确说明：
- "使用 Prompt 工作流，开发 XXX"
- "使用代码工作流，开发 XXX"

### Q6：如何禁用快速跳过？
**A**：修改 `.kiro/kiro.yaml`：
```yaml
workflow:
  quick_skip:
    enabled: false
```
