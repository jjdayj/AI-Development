# 双轨工作流方案设计

**方案创建时间**: 2026-03-02  
**方案版本**: v1.0  
**核心理念**: 针对不同项目类型，使用不同的开发工作流

---

## 🎯 核心洞察

### 问题识别
VibCoding 流程（Phase 1-4，包含大量测试和迭代）适合实际代码项目，但对于 Prompt 基座项目来说：
- ❌ Phase 2-3 的测试和迭代过于繁重
- ❌ Prompt 项目主要是文档和规则，不需要频繁的代码测试
- ❌ Prompt 项目更注重需求理解和规则设计，而不是代码实现

### 解决方案
创建两套工作流：
1. **Prompt 工作流**（轻量级）：适用于 Prompt 基座项目
2. **代码工作流**（完整版）：适用于实际代码项目

---

## 📋 双轨工作流对比

| 维度 | Prompt 工作流 | 代码工作流 |
|------|--------------|-----------|
| **适用场景** | Steering 规则、Hook 规则、文档 | Python 代码、复杂功能 |
| **核心阶段** | Phase 1（探索与决策）+ Phase 4（审查） | Phase 1-4（完整流程） |
| **测试要求** | 轻量级验证（手动测试规则是否生效） | 完整测试（单元测试、集成测试） |
| **迭代方式** | 一次性设计，小幅调整 | 快速原型 → 迭代完善 |
| **文档重点** | 需求探索、规则设计 | 需求探索、架构设计、测试计划 |
| **时间投入** | 1-2 小时 | 4-8 小时 |

## 🔄 工作流 1：Prompt 工作流（轻量级）

### 适用场景
- ✅ Steering 规则开发（如 Git 回滚、VibCoding 流程）
- ✅ Hook 规则开发
- ✅ 文档编写（如开发流程指南）
- ✅ 配置文件修改（如 kiro.yaml）
- ✅ 简单的 Agent 集成功能

### 核心流程

```
Phase 1: 探索与决策（强制执行，60-90 分钟）
   ↓
Phase 2: 规则设计与实现（30-45 分钟）
   ↓
Phase 3: 轻量级验证（15-30 分钟）
   ↓
Phase 4: 审查与发布（15 分钟）

总计：2-3 小时
```

### Phase 1: 探索与决策（强制执行）

#### 🔴 强制检查项（5 个，与代码工作流相同）
1. 目标用户识别
2. 使用场景识别
3. 触发方式识别
4. 交互方式识别
5. 集成方式识别（如果目标用户包含 Agent）

#### 🔴 规则设计决策（Prompt 工作流特有）
```
📐 规则设计检查清单

1. 规则类型：
   - [ ] Steering 规则（.kiro/steering/*.md）
   - [ ] Hook 规则（.kiro/hooks/*.json）
   - [ ] 配置文件（.kiro/kiro.yaml）
   - [ ] 文档（doc/project/*.md）

2. 规则结构：
   - [ ] Front-matter 配置（inclusion、enabled、priority）
   - [ ] 触发条件（关键词、事件）
   - [ ] 执行逻辑（Agent 如何执行）
   - [ ] 错误处理（如何处理异常）
   - [ ] 参考示例（正确 vs 错误）

3. 规则复杂度：
   - [ ] 简单规则（< 200 行，如 Git 回滚）
   - [ ] 中等规则（200-500 行，如 VibCoding 流程）
   - [ ] 复杂规则（> 500 行，需要拆分）

4. 依赖关系：
   - [ ] 是否依赖其他规则？
   - [ ] 是否需要配置文件支持？
   - [ ] 是否需要 Python 脚本支持？
```

#### 输出文档
- `需求探索.md`：5 个强制检查项的答案
- `规则设计决策.md`：规则类型、结构、复杂度、依赖关系

---

### Phase 2: 规则设计与实现（30-45 分钟）

#### 设计原则
- ✅ 一次性设计完整：Prompt 规则不需要迭代，设计好就实现
- ✅ 参考现有规则：参考 Git 回滚、AI Development 等现有规则
- ✅ 保持简洁：避免过度设计，保持规则简洁明了

#### 实现步骤
```
1. 创建规则文件（10 分钟）
   - 创建 .kiro/steering/规则名.md 或 .kiro/hooks/规则名.json
   - 编写 Front-matter 配置

2. 编写核心内容（15-25 分钟）
   - 触发条件
   - 执行逻辑
   - 错误处理
   - 参考示例

3. 编写文档（5-10 分钟）
   - 规则说明
   - 使用示例
   - 注意事项
```

#### 输出文档
- 规则文件：`.kiro/steering/规则名.md` 或 `.kiro/hooks/规则名.json`
- 实施文档：`规则实现说明.md`

---

### Phase 3: 轻量级验证（15-30 分钟）

#### 验证方式
```
🔍 轻量级验证检查清单

1. 规则加载验证：
   - [ ] Agent 是否能加载规则？
   - [ ] Front-matter 配置是否正确？

2. 触发条件验证：
   - [ ] 输入关键词，Agent 是否能识别？
   - [ ] 触发后，Agent 是否执行正确的逻辑？

3. 执行逻辑验证：
   - [ ] Agent 执行的命令是否正确？
   - [ ] Agent 与用户的对话是否自然？

4. 错误处理验证：
   - [ ] 输入错误信息，Agent 是否能正确处理？
   - [ ] 错误提示是否清晰？

验证方式：
- 手动测试（在对话中输入测试用例）
- 不需要编写自动化测试
- 不需要单元测试、集成测试
```

#### 输出文档
- `轻量级验证报告.md`：验证结果、发现的问题、调整建议

---

### Phase 4: 审查与发布（15 分钟）

#### 审查重点
```
📋 审查检查清单

1. 需求符合性：
   - [ ] 是否符合 Phase 1 定义的目标用户？
   - [ ] 是否符合 Phase 1 定义的使用场景？

2. 规则质量：
   - [ ] 规则是否简洁明了？
   - [ ] 是否有参考示例？
   - [ ] 是否有错误处理？

3. 文档完整性：
   - [ ] 是否有规则说明？
   - [ ] 是否有使用示例？
   - [ ] 是否有注意事项？

4. 发布准备：
   - [ ] 规则文件是否在正确的位置？
   - [ ] 配置文件是否更新？
   - [ ] 相关文档是否更新？
```

#### 输出文档
- `审查报告.md`：审查结果、发布清单

## 🔄 工作流 2：代码工作流（完整版）

### 适用场景
- ✅ Python 代码开发（如 AI Development 脚本）
- ✅ 复杂功能开发（如项目管理工具）
- ✅ 需要单元测试、集成测试的功能
- ✅ 需要多次迭代的功能

### 核心流程

```
Phase 1: 探索与决策（强制执行，60-90 分钟）
   ↓
Phase 2: 快速原型（MVP + 强制验证，60-90 分钟）
   ↓
Phase 3: 完善与加固（持续验证，120-180 分钟）
   ↓
Phase 4: 审查与发布（最终验证，30-60 分钟）

总计：4.5-7 小时
```

### Phase 1: 探索与决策（与 Prompt 工作流相同）

#### 🔴 强制检查项（5 个）
1. 目标用户识别
2. 使用场景识别
3. 触发方式识别
4. 交互方式识别
5. 集成方式识别

#### 🔴 架构决策（代码工作流特有）
```
📐 架构决策检查清单

1. 整体架构：
   - [ ] 分层架构（入口层、业务层、数据层）
   - [ ] 模块划分（核心模块、工具模块）
   - [ ] 依赖关系（模块间依赖）

2. Agent 集成层（如果目标用户包含 Agent）：
   - [ ] 调用入口（Steering 规则 / Python 模块）
   - [ ] 参数传递（如何从用户输入中提取参数）
   - [ ] 结果返回（如何将结果展示给用户）

3. 技术选型：
   - [ ] 编程语言（Python / Shell / 其他）
   - [ ] 依赖库（requests / gitpython / 其他）
   - [ ] 配置方式（YAML / JSON / 其他）

4. 测试策略：
   - [ ] 单元测试（pytest / unittest）
   - [ ] 集成测试（端到端测试）
   - [ ] 测试覆盖率（目标 > 80%）
```

#### 输出文档
- `需求探索.md`
- `架构决策.md`
- `需求细化规格说明.md`

---

### Phase 2: 快速原型（MVP + 强制验证）

#### MVP 实现原则
- 只实现核心功能
- 跳过边界情况
- 快速验证想法
- 实现时间：< 2 小时

#### 🔴 强制验证（4 个维度）
1. 目标用户验证
2. 使用场景验证
3. 触发方式验证
4. 交互方式验证

#### 输出文档
- `MVP 实现代码`
- `MVP 测试报告.md`

---

### Phase 3: 完善与加固（持续验证）

#### 实施内容
- 处理边界情况
- 添加错误处理
- 优化性能
- 编写单元测试
- 编写集成测试

#### 🔴 持续验证
- 每次提交前运行测试
- 测试覆盖率 > 80%
- 发现问题立即调整

#### 输出文档
- `完整实现代码`
- `测试代码`
- `测试报告.md`

---

### Phase 4: 审查与发布（最终验证）

#### 审查重点
- 代码质量（代码规范、注释）
- 测试覆盖率（> 80%）
- 文档完整性（README、API 文档）
- 需求符合性（是否符合 Phase 1 定义）

#### 输出文档
- `代码审查报告.md`
- `发布清单.md`

## 🎯 实施方案：如何区分两套工作流？

### 方案 1：通过项目类型自动识别（推荐）

#### 实现方式
在 `.kiro/kiro.yaml` 中配置项目类型：

```yaml
# 项目类型配置
project:
  type: "prompt-base"  # 或 "code-project"
  
# 工作流配置
workflow:
  prompt_workflow:
    enabled: true
    phases: ["phase1", "phase2-lite", "phase3-lite", "phase4"]
  
  code_workflow:
    enabled: false
    phases: ["phase1", "phase2", "phase3", "phase4"]
```

#### Steering 规则
创建 `.kiro/steering/workflow_selector.md`：

```markdown
---
inclusion: always
description: 工作流选择器（根据项目类型自动选择工作流）
priority: highest
---

# 工作流选择器

## 触发条件
当用户提出新功能开发需求时，自动触发。

## 执行逻辑

### 步骤 1：读取项目类型
从 `.kiro/kiro.yaml` 读取 `project.type`：
- 如果是 `prompt-base` → 使用 Prompt 工作流
- 如果是 `code-project` → 使用代码工作流

### 步骤 2：加载对应工作流
- Prompt 工作流 → 加载 `.kiro/steering/prompt_workflow.md`
- 代码工作流 → 加载 `.kiro/steering/code_workflow.md`

### 步骤 3：提示用户
```
✅ 已识别项目类型：[prompt-base / code-project]
✅ 已加载工作流：[Prompt 工作流 / 代码工作流]

开始执行 Phase 1：探索与决策
```
```

### 方案 2：通过关键词手动触发

#### 实现方式
用户在需求中明确指定工作流类型：

```
用户："使用 Prompt 工作流，开发 Git 回滚功能"
→ Agent 加载 Prompt 工作流

用户："使用代码工作流，开发项目管理工具"
→ Agent 加载代码工作流
```

#### Steering 规则
创建 `.kiro/steering/workflow_selector.md`：

```markdown
## 触发条件

### 关键词识别
- "使用 Prompt 工作流" / "prompt workflow" → Prompt 工作流
- "使用代码工作流" / "code workflow" → 代码工作流

### 默认行为
如果用户没有明确指定，根据需求类型自动判断：
- 如果是 Steering 规则、Hook 规则、文档 → Prompt 工作流
- 如果是 Python 代码、复杂功能 → 代码工作流
```

---

### 方案 3：通过文件位置自动识别

#### 实现方式
根据文件位置自动判断工作流类型：

```
.kiro/steering/*.md → Prompt 工作流
.kiro/hooks/*.json → Prompt 工作流
doc/project/*.md → Prompt 工作流

.kiro/modules/*/scripts/*.py → 代码工作流
scripts/*.py → 代码工作流
```

#### Steering 规则
创建 `.kiro/steering/workflow_selector.md`：

```markdown
## 执行逻辑

### 步骤 1：分析需求中的文件路径
从用户需求中提取文件路径：
- 如果包含 `.kiro/steering/` → Prompt 工作流
- 如果包含 `.kiro/hooks/` → Prompt 工作流
- 如果包含 `doc/project/` → Prompt 工作流
- 如果包含 `.py` → 代码工作流

### 步骤 2：如果无法判断，询问用户
```
🤔 无法自动判断工作流类型，请选择：

A. Prompt 工作流（轻量级，适合 Steering 规则、Hook 规则、文档）
B. 代码工作流（完整版，适合 Python 代码、复杂功能）

请选择：A / B
```
```

## 📊 推荐方案对比

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| 方案 1：项目类型自动识别 | ✅ 自动化程度高<br>✅ 不需要用户手动指定 | ⚠️ 需要配置 kiro.yaml<br>⚠️ 切换项目类型需要修改配置 | ⭐⭐⭐⭐⭐ |
| 方案 2：关键词手动触发 | ✅ 灵活性高<br>✅ 用户可以明确指定 | ⚠️ 用户需要记住关键词<br>⚠️ 容易忘记指定 | ⭐⭐⭐ |
| 方案 3：文件位置自动识别 | ✅ 自动化程度高<br>✅ 不需要配置 | ⚠️ 可能判断错误<br>⚠️ 需要用户在需求中提到文件路径 | ⭐⭐⭐⭐ |

## 🎯 最终推荐：方案 1 + 方案 2 组合

### 组合方式
1. **默认行为**：根据 `kiro.yaml` 中的 `project.type` 自动选择工作流
2. **手动覆盖**：用户可以通过关键词手动指定工作流
3. **智能提示**：如果 Agent 不确定，主动询问用户

### 实现步骤

#### 步骤 1：配置 kiro.yaml（5 分钟）
```yaml
# 项目类型配置
project:
  type: "prompt-base"  # 当前是 Prompt 基座项目
  
# 工作流配置
workflow:
  default: "prompt"  # 默认使用 Prompt 工作流
```

#### 步骤 2：创建工作流选择器（30 分钟）
创建 `.kiro/steering/workflow_selector.md`

#### 步骤 3：创建 Prompt 工作流规则（1 小时）
创建 `.kiro/steering/prompt_workflow.md`

#### 步骤 4：创建代码工作流规则（1 小时）
创建 `.kiro/steering/code_workflow.md`

#### 步骤 5：测试验证（30 分钟）
- 测试 Prompt 工作流
- 测试代码工作流
- 测试手动切换

**总工作量**：3 小时
