## 🎯 这是什么？

双轨工作流是 Prompt 基座项目的一个核心功能模块，它会根据你要做的事情，自动选择合适的工作方式：

- � **写规则、写文档**？用轻量级的 Prompt 工作流（2-3 小时）
- 💻 **写代码、开发功能**？用完整的代码工作流（5-8 小时）
- ⚡ **简单修改**？直接跳过流程，快速搞定（5-10 分钟）

就像开车一样，高速公路用高速模式，市区用市区模式，停车场用停车模式。这个功能会自动帮你切换，你也可以手动选择。

## ✨ 核心特性

### 特性 1：智能识别工作流 ⭐⭐⭐⭐⭐

**以前的做法（不够灵活）**：
```yaml
project:
  type: "prompt-base"  # 整个项目只能选一种模式
```

**现在的做法（智能切换）**：
```yaml
workflow:
  default: "prompt"  # 默认模式
  
  # 按文件路径自动切换
  auto_switch:
    - pattern: ".kiro/steering/*.md"
      workflow: "prompt"
    - pattern: "**/*.py"
      workflow: "code"
    - pattern: "doc/project/*.md"
      workflow: "prompt"
```

**实际效果**：
```
你说："修改 .kiro/steering/git_rollback.md"
→ 自动识别：这是 Steering 规则文件
→ 自动选择：Prompt 工作流（轻量级）

你说："开发 scripts/data_processor.py"
→ 自动识别：这是 Python 代码
→ 自动选择：代码工作流（完整版）
```

**优势**：
- ✅ 不用手动切换配置
- ✅ 同一个项目可以同时开发规则和代码
- ✅ 按文件类型自动识别
- ✅ 支持通配符（`**/*.py` 匹配所有 Python 文件）

### 特性 2：快速跳过机制 ⭐⭐⭐⭐⭐

**问题**：改一行配置也要走 2-3 小时的完整流程？太浪费时间了！

**解决方案**：快速跳过机制

**配置示例**：
```yaml
workflow:
  quick_skip:
    enabled: true
    
    # 手动跳过关键词
    manual_skip_keywords:
      - "快速修改"
      - "简单修改"
      - "跳过流程"
    
    # 自动跳过条件
    auto_skip_conditions:
      - type: "line_count"
        threshold: 10
        description: "修改行数 < 10 行"
      
      - type: "file_type"
        patterns: ["*.yaml", "*.json", "*.txt"]
        description: "简单配置文件"
      
      - type: "change_type"
        types: ["typo", "format", "comment"]
        description: "错别字、格式化、注释修改"
```

**使用示例**：

**场景 1：手动跳过**
```
你说："快速修改 kiro.yaml，添加一个配置项"

Kiro：
🤔 检测到快速修改关键词

这是一个简单修改吗？可以跳过完整的工作流吗？

适合跳过的情况：
- 简单配置修改（< 10 行）
- 纯文档错别字修正
- 简单的规则微调（不改变核心逻辑）
- 格式化、注释修改

请确认：
A. 是，跳过流程（直接执行修改，5-10 分钟）
B. 否，执行完整流程（Phase 1-5，2.5-8 小时）

你说："A"

Kiro：
✅ 已跳过完整工作流
直接执行修改...
```

**场景 2：自动检测**
```
你说："修改 README.md 的错别字"

Kiro：
🤔 检测到可能是简单修改

根据你的需求描述，这可能是一个简单修改：
- 检测到关键词："错别字"
- 文件类型：*.md（文档文件）

是否跳过完整的工作流？

请确认：
A. 是，跳过流程（5-10 分钟）
B. 否，执行完整流程（2.5-8 小时）
```

**优势**：
- ✅ 简单修改不走完整流程
- ✅ 节省时间（从 2-3 小时降到 5-10 分钟）
- ✅ 用户确认机制（避免误跳过）
- ✅ 支持手动和自动两种方式

### 特性 3：两套工作流对比 ⭐⭐⭐⭐

**Prompt 工作流（轻量级）**：
- 适合：Steering 规则、Hook 规则、文档
- 时间：2.5-3.5 小时
- 测试：手动测试即可
- 流程：探索 → 设计 → 验证 → 发布 → 总结

**代码工作流（完整版）**：
- 适合：Python 代码、复杂功能
- 时间：5-8 小时
- 测试：单元测试 + 集成测试（覆盖率 > 80%）
- 流程：探索 → 原型 → 完善 → 发布 → 总结

**对比表格**：

| 维度 | Prompt 工作流 | 代码工作流 |
|------|--------------|-----------|
| **适用场景** | 规则、文档 | 代码、功能 |
| **时间投入** | 2.5-3.5 小时 | 5-8 小时 |
| **测试要求** | 手动测试 | 自动化测试 |
| **复杂度** | 轻量级 | 完整版 |

---

### 特性 4：强制总结反思 ⭐⭐⭐⭐⭐

**问题**：项目做完就忘了，下次还犯同样的错误。

**解决方案**：每个项目结束后，强制执行总结反思（Phase 5）

**总结内容**：
1. 🔴 遇到了哪些问题？
2. 🔴 根本原因是什么？（5 Why 分析法）
3. 🔴 学到了什么经验教训？
4. 🔴 下次怎么改进？
5. 🔴 知识如何沉淀？

**输出文档**：
```
doc/project/
└── YYYYMMDD_项目名-总结与反思.md
    - 项目背景和结果
    - 问题识别与分类
    - 根本原因分析
    - 经验教训
    - 改进措施
    - 知识沉淀
```

**优势**：
- ✅ 避免重复犯错
- ✅ 形成知识积累
- ✅ 团队经验共享
- ✅ 持续改进流程

### 特性 5：防止需求理解错误 ⭐⭐⭐⭐⭐

**问题**：开发了半天，发现理解错了需求，白干了。

**解决方案**：Phase 1 的 5 个强制检查项

**5 个必答问题**：
1. 🔴 目标用户是谁？（Agent / 人类用户 / 第三方应用）
2. 🔴 使用场景是什么？（Agent 内部调用 / 独立工具 / API 服务）
3. 🔴 如何触发？（Steering 规则 / 命令行 / API）
4. 🔴 如何交互？（Agent 对话 / 命令行交互 / 无交互）
5. 🔴 如何集成？（如果目标用户包含 Agent）

**关键规则**：
- Agent 遇到不确定信息时，必须主动提问
- 不得自行假设
- 必须得到明确答案才能继续

**案例对比**：

**❌ 错误案例：Git 提交工作流（已回滚）**
```
错误理解：将 Agent 内部调用理解成独立工具
根本原因：Phase 1 没有明确目标用户
结果：开发了一个独立的 Python 脚本，但实际需要的是 Steering 规则
教训：必须在需求探索阶段强制回答 5 个问题
```

**✅ 正确案例：Git 回滚功能**
```
目标用户：Agent
使用场景：Agent 内部调用
触发方式：Steering 规则识别关键词
实现方式：Steering 规则 + git 命令
工作流：Prompt 工作流
结果：成功实现，符合预期
```

**优势**：
- ✅ 避免需求理解错误
- ✅ 减少返工
- ✅ 提高开发效率
- ✅ 明确项目方向

## 🚀 完整工作流程

### 流程概览

```
用户提出需求
    ↓
【步骤 1】快速跳过判断
    ├─ 是简单修改？→ 快速模式（5-10 分钟）
    └─ 否 → 继续
    ↓
【步骤 2】识别工作流类型
    ├─ 规则/文档 → Prompt 工作流
    └─ 代码/功能 → 代码工作流
    ↓
【步骤 3】执行 5 个阶段
    ├─ Phase 1: 探索与决策（60-90 分钟）
    ├─ Phase 2: 实现（30-90 分钟）
    ├─ Phase 3: 验证/完善（15-180 分钟）
    ├─ Phase 4: 审查与发布（15-60 分钟）
    └─ Phase 5: 总结与反思（30-60 分钟）
```

---

### Phase 1：探索与决策（60-90 分钟）

**这个阶段做什么？**
- 搞清楚要做什么
- 搞清楚给谁用
- 搞清楚怎么用
- 避免理解错需求

**5 个必答问题**：
1. 目标用户是谁？
2. 使用场景是什么？
3. 如何触发？
4. 如何交互？
5. 如何集成？

**关键规则**：
- Agent 不确定就问，不能猜
- 必须得到明确答案才能继续

---

### Phase 2：实现

**Prompt 工作流（30-45 分钟）**：
- 一次性设计完整的规则
- 参考现有规则的写法
- 保持简洁，不要过度设计

**代码工作流（60-90 分钟）**：
- 快速实现 MVP（最小可用版本）
- 先跑通核心功能
- 不追求完美，快速迭代

---

### Phase 3：验证/完善

**Prompt 工作流（15-30 分钟）**：
- 手动测试规则是否生效
- 测试几个典型场景即可
- 不需要写自动化测试

**代码工作流（120-180 分钟）**：
- 处理边界情况和异常
- 写单元测试和集成测试
- 测试覆盖率要达到 80% 以上

---

### Phase 4：审查与发布（15-60 分钟）

**检查清单**：
- ✅ 是否符合 Phase 1 的定义？
- ✅ 代码/规则质量是否合格？
- ✅ 文档是否完整？
- ✅ 是否可以发布？

---

### Phase 5：总结与反思（30-60 分钟）⭐

**这个阶段做什么？**
- 回顾项目过程
- 识别遇到的问题
- 分析根本原因
- 提取经验教训
- 制定改进措施

**5 个强制项**：
1. 问题识别与分类
2. 根本原因分析（5 Why）
3. 经验教训提取
4. 改进措施制定
5. 知识沉淀

**输出文档**：
```
doc/project/
└── YYYYMMDD_项目名-总结与反思.md
```

## ⚙️ 配置说明

### 配置文件位置
```
.kiro/kiro.yaml
```

### 完整配置示例
```yaml
workflow:
  # 默认工作流
  default: "prompt"
  
  # 按文件路径自动切换
  auto_switch:
    # Prompt 工作流
    - pattern: ".kiro/steering/*.md"
      workflow: "prompt"
    - pattern: ".kiro/hooks/*.json"
      workflow: "prompt"
    - pattern: "doc/project/*.md"
      workflow: "prompt"
    
    # 代码工作流
    - pattern: "**/*.py"
      workflow: "code"
    - pattern: "scripts/*.py"
      workflow: "code"
  
  # 快速跳过配置
  quick_skip:
    enabled: true
    
    # 手动跳过关键词
    manual_skip_keywords:
      - "快速修改"
      - "简单修改"
      - "跳过流程"
      - "quick fix"
      - "simple change"
    
    # 自动跳过条件
    auto_skip_conditions:
      - type: "line_count"
        threshold: 10
        description: "修改行数 < 10 行"
      
      - type: "file_type"
        patterns: ["*.yaml", "*.json", "*.txt"]
        description: "简单配置文件"
      
      - type: "change_type"
        types: ["typo", "format", "comment"]
        description: "错别字、格式化、注释修改"
```

## 🎯 常见问题

### Q1：什么时候用 Prompt 工作流？
**A**：开发 Steering 规则、Hook 规则、文档时使用。特点是轻量级，不需要写自动化测试。

### Q2：什么时候用代码工作流？
**A**：开发 Python 代码、复杂功能时使用。特点是完整版，需要写单元测试和集成测试。

### Q3：什么时候可以快速跳过？
**A**：简单修改时可以跳过，比如：
- 修改配置文件的一个参数
- 修正文档的错别字
- 简单的格式化调整
- 修改注释

### Q4：Phase 5 可以跳过吗？
**A**：不可以。Phase 5 是强制执行的，用于总结经验教训，避免重复犯错。

### Q5：如何手动指定工作流？
**A**：在需求中明确说明：
- "使用 Prompt 工作流，开发 XXX"
- "使用代码工作流，开发 XXX"

### Q6：如何禁用快速跳过？
**A**：修改 `.kiro/kiro.yaml`：
```yaml
workflow:
  quick_skip:
    enabled: false
```
