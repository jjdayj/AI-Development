## 🎨 核心设计

### 工作流选择机制
```
用户输入触发关键词
    ↓
强制询问用户选择（不可跳过）
    ├─ A. 原生功能开发流程（4 阶段，2 小时）
    ├─ B. 二次开发流程（5 阶段，3 小时）
    └─ C. 不进入工作流，正常作答
    ↓
路由到对应流程
```

### 原生开发流程（4 阶段）
```
Phase 1：了解项目与需求
    ↓
    - 确认需求边界
    - 扫查可复用资产
    - 创建目录结构（requirements/{module}/{version}/）
    ↓
Phase 2：分析需求
    ↓
    - 拆解为最小可执行单元
    - 主动澄清所有不确定点
    - 询问：测试需求、文档需求、GUI 偏好
    ↓
Phase 3：制定执行计划
    ↓
    - 按"改一点测一点"拆分步骤
    - 对齐 kiro 规范
    - 根据 Phase 2 的选择调整计划
    ↓
Phase 4：落地执行
    ↓
    - 执行单个最小单元
    - 立即测试（如果需要）
    - 测试通过→下一个；不通过→回滚修复
    ↓
Phase 5：总结（可选）
    ↓
    - 生成总结文档（如果需要）
    - 更新 current_context.yaml
```

### 二次开发流程（5 阶段）
```
Phase 0：评估相似代码
    ↓
    - 分析适配性（目录、状态、GUI、依赖）
    - 评估改造成本（低/中/高/建议重写）
    - 提取可复用片段
    - 确认改造风险
    - 询问：继续二次开发 / 改为原生开发 / 取消
    ↓
Phase 1-4：复用原生开发流程
    ↓
    - Phase 1：将相似代码复用计划加入资产清单
    - Phase 3：将代码改造拆分为最小单元
    - Phase 4：新增兼容性测试
    ↓
Phase 5：总结（可选）
```

## 🔑 关键特性

### 1. 轻量化
- 原生开发仅 4 个阶段（vs 旧版 5 个阶段）
- 极简输出，不要冗长文档
- 测试和文档都是可选的

### 2. 改一点测一点
- Phase 3 按最小可测试单元拆分步骤
- Phase 4 每个单元完成后立即测试
- 测试不通过仅回滚当前单元，不影响已完成模块

### 3. 主动澄清
- Phase 2 强制主动提问
- 必问：功能冲突点
- 条件问：GUI 偏好、测试需求、文档需求
- 不猜测，所有不确定点都要澄清

### 4. 支持二次开发
- Phase 0 评估现有代码
- 分析适配性、改造成本、风险
- 给出明确建议：继续二次开发 / 改为原生开发
- 新增兼容性测试

### 5. 保留核心资产
- ✅ current_context.yaml 状态管理
- ✅ requirements/{module}/{version}/ 目录规范
- ✅ meta.yaml 元信息管理
- ✅ Git 提交自动化联动

## 🎯 设计决策

### 决策 1：强制询问用户选择
**原因**：避免 AI 自动判断错误，让用户明确选择工作流类型

**实现**：
- 检测到触发关键词后，必须询问用户
- 提供 3 个选项：A（原生）/ B（二次开发）/ C（不进入工作流）
- 每个选项都有清晰的说明和预计时间

### 决策 2：测试和文档可选
**原因**：单人开发不需要强制测试和文档，根据需求灵活配置

**实现**：
- Phase 2 询问用户是否需要测试和文档
- 根据用户选择调整 Phase 3 和 Phase 4 的计划
- Phase 5 根据选择决定是否执行

### 决策 3：改一点测一点
**原因**：体现 VibeCoding 的核心理念，小步快跑

**实现**：
- Phase 3 按最小可测试单元拆分步骤
- Phase 4 每个单元完成后立即测试
- 测试不通过仅回滚当前单元

### 决策 4：支持二次开发
**原因**：用户经常需要基于现有代码进行改造

**实现**：
- 新增 Phase 0 评估阶段
- 分析适配性、改造成本、风险
- 给出明确建议
- 新增兼容性测试

### 决策 5：保留核心资产
**原因**：状态管理和目录规范是项目的核心价值

**实现**：
- Phase 1 创建 requirements/{module}/{version}/ 目录
- Phase 1 生成 meta.yaml 和 requirement.md
- Phase 1 更新 current_context.yaml
- Phase 5 更新 current_context.yaml（标记完成）

## 🐛 潜在问题和解决方案

### 问题 1：用户可能不知道选择哪个工作流
**解决方案**：
- 在询问时提供清晰的适用场景说明
- 提供预计时间参考
- 如果用户选错，可以随时切换

### 问题 2：Phase 2 的问题可能不够全面
**解决方案**：
- 在 config.yaml 中定义标准问题模板
- 根据需求类型（是否涉及 GUI）动态调整问题
- 鼓励用户补充其他不确定点

### 问题 3：二次开发的评估可能不准确
**解决方案**：
- Phase 0 提供详细的评估维度（适配性、成本、风险）
- 给出明确的建议（继续 / 改为原生 / 取消）
- 用户可以根据评估结果自主决策

VibeCoding 工作流 v2.0 成功实现了以下目标：

新工作流完全替代了旧版本（v1.0 系列），更适合单人开发和快速迭代场景。

## 🔄 v2.0.1 更新（2026-03-04）

### 新增功能：需求复杂度分析

在用户选择工作流类型（A 或 B）后，新增了一个强制执行的"需求复杂度分析"阶段。

### 更新内容

#### 1. 新增需求复杂度分析阶段
- **触发条件**：用户选择 A（原生功能开发）或 B（二次开发）
- **执行时机**：在选择工作流后，进入具体流程前
- **语言**：所有询问和分析都使用中文

#### 2. 四种复杂度情况

**情况 A：简单清晰**
- 功能点 ≤ 3 个，文件 ≤ 5 个
- 直接进入工作流，无需生成探索文档

**情况 B：较复杂但单一需求**
- 功能点 4-8 个，文件 6-15 个
- 建议生成需求探索文档（询问用户）
- 方便后续对话追踪

**情况 C：复杂需拆分**
- 功能点 > 8 个，文件 > 15 个
- 建议生成需求探索文档并拆分为多个子需求
- 记录子需求之间的依赖关系

**情况 D：过于复杂**
- 功能点 > 15 个，文件 > 30 个
- 通知用户不适合 VibeCoding 工作流
- 建议进一步拆分或使用其他方式

#### 3. 需求探索文档

**文档路径**：`requirements/exploration/{需求名称}_exploration.md`

**文档内容**：
- 需求概述
- 功能点列表
- 不确定点列表
- 技术方案初步思路
- 子需求列表（如果需要拆分）
- 依赖关系图（如果需要拆分）
- 开发顺序建议（如果需要拆分）
- 后续对话追踪记录

#### 4. 更新的流程图

```
用户输入触发关键词
    ↓
询问选择工作流类型（中文）
    ├─ A. 原生功能开发
    ├─ B. 二次开发
    └─ C. 不进入工作流
    ↓
需求复杂度分析（仅 A/B，中文）
    ├─ 简单：直接进入工作流
    ├─ 中等：建议生成探索文档
    ├─ 复杂：建议拆分需求
    └─ 过于复杂：通知用户
    ↓
进入对应工作流
```

### 设计理由

#### 为什么需要需求复杂度分析？

1. **避免需求过于复杂**：VibeCoding 工作流设计用于 2-3 小时的快速开发，过于复杂的需求不适合
2. **提前发现问题**：在进入具体流程前，提前发现需求的复杂度问题
3. **引导需求拆分**：对于复杂需求，引导用户拆分为多个子需求
4. **方便对话追踪**：对于需要多次对话的需求，生成探索文档方便追踪

#### 为什么使用中文？

1. **用户友好**：中文更直观，用户更容易理解
2. **减少误解**：避免因语言问题导致的误解
3. **符合习惯**：用户在日常工作中使用中文

### 配置更新

在 `config.yaml` 中新增了复杂度分析的标准：

complexity_analysis:

    function_points: 15
    files: 30
