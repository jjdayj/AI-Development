# Q06: 如果模块 A 依赖模块 B，但模块 B 被禁用了，会发生什么？

**提问时间**: 2026-03-03

## 问题

如果模块 A 依赖模块 B，但模块 B 被禁用了，会发生什么？

## 回答

根据 `main.md` 文档，如果模块 A 依赖模块 B，但模块 B 被禁用，系统会进行依赖校验并阻止模块 A 加载。

## 依赖校验机制（步骤 6）

### 步骤 6：校验依赖关系

```
检查每个激活模块的依赖：
1. 确保所有依赖的模块都在激活列表中
2. 检测循环依赖（A 依赖 B，B 依赖 A）
3. 依赖不满足的模块不应该被加载

错误处理：
- 缺失依赖：输出警告，跳过该模块
- 循环依赖：输出错误，跳过所有涉及的模块
```

---

## 具体场景分析

### 场景：模块 A 依赖模块 B，但 B 被禁用

**配置示例**：

```yaml
# .kiro/config.yaml
modules:
  module-a:
    enabled: true    # A 启用
    version: v1.0
    priority: 100
  
  module-b:
    enabled: false   # B 禁用 ❌
    version: v1.0
    priority: 90
```

```yaml
# .kiro/modules/module-a/v1.0/config.yaml
activation_conditions:
  always: true

dependencies:
  - module-b    # A 依赖 B
```

---

### 处理流程

**步骤 1：读取顶层配置**
```
读取结果：
- module-a: enabled=true
- module-b: enabled=false
```

**步骤 2：筛选启用的模块**
```
筛选结果：
- module-a ✓ (enabled=true)
- module-b ✗ (enabled=false，被过滤掉)

启用列表：[module-a]
```

**步骤 3-5：读取配置、合并、计算激活状态**
```
module-a:
  - 配置读取成功
  - 激活条件：always=true
  - 依赖：[module-b]
  - 激活状态：true
```

**步骤 6：校验依赖关系** ⚠️
```
检查 module-a 的依赖：
- 需要依赖：[module-b]
- 激活列表：[module-a]
- 结果：module-b 不在激活列表中 ❌

处理：
- 输出警告日志
- 跳过 module-a
- module-a 不会被加载
```

## 解决方案

### 方案 1：启用被依赖的模块

```yaml
# .kiro/config.yaml
modules:
  module-a:
    enabled: true
    version: v1.0
    priority: 100
  
  module-b:
    enabled: true   # 改为 true ✓
    version: v1.0
    priority: 90
```

**结果**：module-a 和 module-b 都会被加载

---

### 方案 2：移除依赖关系

```yaml
# .kiro/modules/module-a/v1.0/config.yaml
activation_conditions:
  always: true

dependencies: []  # 移除对 module-b 的依赖
```

**结果**：module-a 可以独立加载

---

## 依赖校验规则总结

### 规则 1：依赖必须在激活列表中

```
依赖校验 = 所有依赖模块都在激活列表中
```

**激活列表的条件**：
1. `enabled: true`（全局开关）
2. 激活条件匹配（activation_conditions）
3. 依赖校验通过

---

### 规则 2：缺失依赖导致跳过

```
如果依赖缺失：
- 输出警告日志
- 跳过该模块
- 不会加载 Steering 文件
```

---

### 规则 3：依赖链传播

```
如果 A → B → C，且 C 被禁用：
1. C 被禁用，不在激活列表
2. B 依赖 C，校验失败，被跳过
3. A 依赖 B，B 已被跳过，A 也被跳过
```

---

## 最佳实践

### 1. 明确依赖关系

在模块文档中清晰说明依赖：

```yaml
# .kiro/modules/module-a/v1.0/README.md
## 依赖

本模块依赖以下模块：
- module-b (v1.0+)：提供基础功能

如果 module-b 被禁用，本模块将无法加载。
```

---

### 2. 最小化依赖

尽量减少模块间的依赖关系：

```yaml
# 不推荐：过多依赖
dependencies:
  - module-a
  - module-b
  - module-c
  - module-d

# 推荐：最小依赖
dependencies:
  - module-a  # 只依赖核心模块
```

---

### 3. 核心模块无依赖

核心模块应该独立，不依赖其他模块：

```yaml
# .kiro/modules/workflow/v1.0/config.yaml
dependencies: []  # 核心模块无依赖
```

---

## 总结

**如果模块 A 依赖模块 B，但 B 被禁用**：

1. ✗ 模块 B 不会进入启用列表（步骤 2）
2. ✗ 模块 A 的依赖校验失败（步骤 6）
3. ⚠️ 系统输出警告日志
4. ✗ 模块 A 被跳过，不会加载
5. ✓ 系统继续处理其他模块（不会崩溃）

**关键点**：
- 依赖是强制性的，不是可选的
- 依赖缺失会阻止模块加载
- 依赖链会传播失败
- 系统会优雅降级，不会崩溃
