# Q09: 顶层配置和模块配置冲突时，哪个优先？

**提问时间**: 2026-03-03

## 问题

顶层配置和模块配置冲突时，哪个优先？

## 回答

根据 `main.md` 文档，配置冲突时有明确的优先级规则。

## 配置优先级规则（步骤 4）

### 步骤 4：合并配置

```
将顶层配置和模块配置合并，遵循以下规则：
- 顶层配置优先级更高
- 模块配置作为补充
- 嵌套字典进行深度合并
```

**核心原则**：
```
顶层配置 > 模块配置
```

---

## 合并规则详解

### 规则 1：顶层配置优先

当同一个字段在两个配置中都存在时，使用顶层配置的值。

**示例**：
```yaml
# 顶层配置 (.kiro/config.yaml)
modules:
  workflow:
    enabled: true
    version: v1.0
    priority: 200

# 模块配置 (.kiro/modules/workflow/v1.0/config.yaml)
priority: 100  # 这个会被忽略
```

**合并结果**：
```yaml
enabled: true
version: v1.0
priority: 200  # 使用顶层配置的值
```

---

### 规则 2：模块配置作为补充

当字段只在模块配置中存在时，使用模块配置的值。

**示例**：
```yaml
# 顶层配置 (.kiro/config.yaml)
modules:
  workflow:
    enabled: true
    version: v1.0
    priority: 200

# 模块配置 (.kiro/modules/workflow/v1.0/config.yaml)
activation_conditions:
  always: true
dependencies: []
module_specific_config:
  custom_field: value
```

**合并结果**：
```yaml
enabled: true                    # 来自顶层配置
version: v1.0                    # 来自顶层配置
priority: 200                    # 来自顶层配置
activation_conditions:           # 来自模块配置（补充）
  always: true
dependencies: []                 # 来自模块配置（补充）
module_specific_config:          # 来自模块配置（补充）
  custom_field: value
```

---

### 规则 3：嵌套字典深度合并

对于嵌套的字典结构，进行深度合并。

**示例**：
```yaml
# 顶层配置 (.kiro/config.yaml)
modules:
  workflow:
    enabled: true
    version: v1.0
    priority: 200
    custom_settings:
      field_a: value_from_top
      field_b: value_from_top

# 模块配置 (.kiro/modules/workflow/v1.0/config.yaml)
custom_settings:
  field_b: value_from_module  # 冲突
  field_c: value_from_module  # 补充
```

**合并结果**：
```yaml
enabled: true
version: v1.0
priority: 200
custom_settings:
  field_a: value_from_top      # 来自顶层配置
  field_b: value_from_top      # 冲突，使用顶层配置
  field_c: value_from_module   # 来自模块配置（补充）
```

### 示例 2：冲突场景

**顶层配置**：
```yaml
# .kiro/config.yaml
modules:
  ai-dev:
    enabled: true
    version: v1.0
    priority: 100
    custom_config:
      auto_commit: false      # 顶层设置
      timeout: 300            # 顶层设置
```

**模块配置**：
```yaml
# .kiro/modules/ai-dev/v1.0/config.yaml
priority: 150                 # 冲突：模块想要更高优先级

activation_conditions:
  always: true

custom_config:
  auto_commit: true           # 冲突：模块想要自动提交
  max_retries: 3              # 补充：新字段
```

**合并结果**：
```yaml
enabled: true
version: v1.0
priority: 100                 # 使用顶层配置（顶层优先）
activation_conditions:
  always: true
custom_config:
  auto_commit: false          # 使用顶层配置（顶层优先）
  timeout: 300                # 来自顶层配置
  max_retries: 3              # 来自模块配置（补充）
```

**关键点**：
- `priority`: 100（顶层）而非 150（模块）
- `auto_commit`: false（顶层）而非 true（模块）
- `max_retries`: 3（模块补充）

## 合并算法实现

### Python 实现示例

```python
def merge_configs(top_config, module_config):
    """
    合并顶层配置和模块配置
    
    规则：
    1. 顶层配置优先级更高
    2. 模块配置作为补充
    3. 嵌套字典进行深度合并
    """
    result = {}
    
    # 先添加模块配置的所有字段
    for key, value in module_config.items():
        result[key] = value
    
    # 然后用顶层配置覆盖（深度合并）
    for key, value in top_config.items():
        if key in result and isinstance(value, dict) and isinstance(result[key], dict):
            # 嵌套字典：深度合并
            result[key] = merge_configs(value, result[key])
        else:
            # 非字典或不存在：直接覆盖
            result[key] = value
    
    return result
```

**使用示例**：
```python
top_config = {
    'enabled': True,
    'priority': 200,
    'settings': {
        'field_a': 'top_value',
        'field_b': 'top_value'
    }
}

module_config = {
    'priority': 100,  # 会被覆盖
    'activation_conditions': {'always': True},  # 补充
    'settings': {
        'field_b': 'module_value',  # 会被覆盖
        'field_c': 'module_value'   # 补充
    }
}

result = merge_configs(top_config, module_config)

# 结果：
# {
#     'enabled': True,                          # 来自顶层
#     'priority': 200,                          # 顶层覆盖模块
#     'activation_conditions': {'always': True}, # 来自模块
#     'settings': {
#         'field_a': 'top_value',               # 来自顶层
#         'field_b': 'top_value',               # 顶层覆盖模块
#         'field_c': 'module_value'             # 来自模块
#     }
# }
```

## 为什么顶层配置优先？

### 原因 1：全局控制

顶层配置是项目级别的全局配置，应该有最终决定权。

**使用场景**：
```yaml
# 项目管理员在顶层配置中禁用某个模块
modules:
  experimental-feature:
    enabled: false  # 全局禁用

# 即使模块配置想要启用，也会被顶层配置覆盖
```

---

### 原因 2：环境适配

不同环境可能需要不同的配置，顶层配置可以灵活调整。

**示例**：
```yaml
# 开发环境：启用详细日志
modules:
  ai-dev:
    enabled: true
    settings:
      verbose_logging: true

# 生产环境：禁用详细日志
modules:
  ai-dev:
    enabled: true
    settings:
      verbose_logging: false
```

---

### 原因 3：安全控制

顶层配置可以覆盖模块的不安全设置。

**示例**：
```yaml
# 模块配置想要自动提交
# .kiro/modules/git-commit/v1.0/config.yaml
settings:
  auto_commit: true

# 顶层配置禁用自动提交（安全考虑）
# .kiro/config.yaml
modules:
  git-commit:
    settings:
      auto_commit: false  # 覆盖模块配置
```

---

### 原因 4：优先级调整

顶层配置可以调整模块的优先级。

**示例**：
```yaml
# 模块配置的默认优先级
# .kiro/modules/workflow/v1.0/config.yaml
priority: 100

# 顶层配置提升优先级
# .kiro/config.yaml
modules:
  workflow:
    priority: 200  # 覆盖模块配置
```

## 最佳实践

### 1. 顶层配置管理全局开关

```yaml
# .kiro/config.yaml
modules:
  workflow:
    enabled: true      # 全局开关
    version: v1.0      # 版本控制
    priority: 200      # 优先级控制
```

---

### 2. 模块配置管理功能细节

```yaml
# .kiro/modules/workflow/v1.0/config.yaml
activation_conditions:
  always: true

dependencies: []

module_specific_config:
  default_workflow: "prompt"
  quick_skip_enabled: true
  phase5_mandatory: true
```

---

### 3. 使用顶层配置覆盖特定设置

```yaml
# .kiro/config.yaml
modules:
  ai-dev:
    enabled: true
    version: v1.0
    priority: 100
    # 覆盖模块的特定设置
    module_specific_config:
      auto_commit: false  # 禁用自动提交
      timeout: 600        # 增加超时时间
```

## 总结

**配置优先级规则**：
```
顶层配置 > 模块配置
```

**合并规则**：
1. 顶层配置优先级更高（冲突时使用顶层配置）
2. 模块配置作为补充（顶层没有的字段使用模块配置）
3. 嵌套字典进行深度合并（递归应用上述规则）

**使用场景**：
- 顶层配置：全局开关、版本控制、优先级、环境适配、安全控制
- 模块配置：激活条件、依赖关系、模块特定功能

**最佳实践**：
- 顶层配置管理全局设置
- 模块配置管理功能细节
- 使用顶层配置覆盖特定设置
- 文档化配置覆盖原因
