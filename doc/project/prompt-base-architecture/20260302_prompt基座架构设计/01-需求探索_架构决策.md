## 一、核心目标

用户明确的项目目标：
1. **商业化**：模块可独立授权和发布
2. **开源**：支持部分模块开源，部分商业化

## 二、核心疑问（需求探索阶段）

### 疑问 1：doc 目录的 Git 管理策略
**问题**：doc_save_rules 生成的 doc 目录，应该纳入 Git 管理还是只存本地？

**讨论结果**：
- Kiro 初始方案：分场景处理，默认不纳入 Git
- 豆包方案：优先纳入 Git，用 .gitignore 精细化过滤
- **最终决策**：采用豆包方案 + 物理分离
  - `doc/personal/`：个人日志，.gitignore 排除
  - `doc/project/`：项目文档，强制纳入 Git

### 疑问 2：.kiro 目录管理混乱问题
**问题**：随着功能增加，.kiro 目录可能很乱，业界有好的解决办法吗？

**讨论结果**：
- Kiro 方案 A：按功能分层（steering/、scripts/、configs/）
- Kiro 方案 B：使用命名前缀
- 豆包方案：按模块+版本分层（ai-dev/v1.0/、doc-save/v1.1/）
- **最终决策**：模块优先结构（Kiro 优化方案）
  ```
  .kiro/modules/
    ├── ai-dev/
    │   ├── v1.0/
    │   │   ├── steering/
    │   │   ├── scripts/
    │   │   └── configs/
    │   ├── v1.1/
    │   ├── LICENSE
    │   └── README.md
  ```

### 疑问 3：提示词版本管理
**问题**：如何版本化管理提示词，便于回滚和效果对比？

**讨论结果**：
- Kiro 方案：current/ + history/vX.X_timestamp/，实际文件存储
- 豆包方案：软链接指向 history，避免文件重复
- **最终决策**：采用 Kiro 方案（实际文件，不用软链接）
  - 原因：跨平台兼容性更好，Windows 软链接需要管理员权限
  - 增强：采纳豆包的"效果评估"字段

### 疑问 4：双 Agent 协作架构
**问题**：如何实现"需求提示词生成 + 代码执行"的双 Agent 协作？

**讨论结果**：
- Kiro 方案：基于 Kiro Hooks 触发
- 豆包方案：独立 Python 脚本（prompt_generator.py + code_executor.py）
- **最终决策**：混合方案
  - 核心逻辑用独立脚本（解耦）
  - 触发入口用 Kiro Hooks（集成）

### 疑问 5：需求演进过程记录
**问题**：如何记录"前置疑问"这种未形成正式需求的讨论？

**讨论结果**：
- Kiro 方案：RFC（Request for Comments）+ ADR（Architecture Decision Record）
- 豆包方案：需求探索（Requirement Exploration）+ ADR
- **最终决策**：采用豆包术语 + Kiro 的详细模板（简化版）
  - `req_exploration/rfcs/`：记录讨论过程
  - `req_exploration/adrs/`：记录最终决策

## 三、架构决策（ADR）

### ADR-001：采用"模块优先"目录结构

**状态**：已采纳（2026-03-02）

**背景**：
- 项目目标包含"商业化 + 开源 + 解耦合"
- 需要支持模块独立发布和授权

**决策**：
采用 `.kiro/modules/{module}/{version}/` 结构，每个模块包含完整的 steering/scripts/configs。

**理由**：
1. ✅ 模块独立发布：可单独打包到 npm/pypi
2. ✅ 独立授权：每个模块独立 LICENSE
3. ✅ 独立文档：每个模块独立 README
4. ✅ CI/CD 友好：可对单个模块独立构建

**后果**：
- 优点：模块完全解耦，商业化友好
- 缺点：目录层级稍深（可接受）

---

### ADR-002：不使用软链接，改用配置文件管理版本

**状态**：已采纳（2026-03-02）

**背景**：
- 豆包方案建议用软链接（latest -> v1.1/）
- 需要考虑跨平台兼容性和商业化场景

**决策**：
使用 `kiro.yaml` 配置文件指定当前激活的模块版本，不使用软链接。

**理由**：
1. ✅ 跨平台：Windows 软链接需要管理员权限
2. ✅ Git 友好：软链接在 Git 中支持不完善
3. ✅ 安全性：配置文件改错可恢复，软链接误删难恢复
4. ✅ 商业化：客户环境可能不支持软链接

**后果**：
- 优点：更安全，更兼容
- 缺点：需要额外的 module_loader.py 脚本（可接受）

---

### ADR-003：doc 目录物理分离（personal/project）

**状态**：已采纳（2026-03-02）

**背景**：
- doc 目录既包含个人日志，也包含项目文档
- 需要区分哪些纳入 Git，哪些排除

**决策**：
将 doc 拆分为 `doc/personal/`（排除）和 `doc/project/`（纳入 Git）。

**理由**：
1. ✅ 清晰：物理分离比 .gitignore 过滤更直观
2. ✅ 安全：避免误提交个人日志
3. ✅ 灵活：个人日志可自由记录，不污染 Git 历史

**后果**：
- 优点：清晰、安全
- 缺点：需要手动区分文档存放位置（可接受）

---

### ADR-004：changelog 增加"效果评估"字段

**状态**：已采纳（2026-03-02）

**背景**：
- 提示词迭代需要评估不同版本的效果
- 豆包方案建议在 changelog 中增加效果评估

**决策**：
在 changelog.md 中增加以下字段：
- 修改内容
- 变更原因
- **效果评估**（新增）
- **测试用例**（新增）
- 关联文件
- 回滚风险

**理由**：
1. ✅ 数据驱动：量化提示词优化效果
2. ✅ 决策支持：便于选择最优版本
3. ✅ 知识沉淀：记录哪些提示词更有效

**后果**：
- 优点：提示词迭代更科学
- 缺点：需要手动填写评估结果（可接受）

## 五、实施计划

### 开发方法论：VibCoding

本项目采用 **VibCoding（振动式编码）** 作为核心开发方法论。

**核心理念**：快速反馈、小步迭代、持续调整

**完整流程**：参见 `16-VibCoding开发流程指南.md`

**4 个阶段**：
1. **Phase 1: 探索与决策**（需求探索 → 架构决策 → 需求细化）
2. **Phase 2: 快速原型**（MVP 实现 → 快速测试 → 反馈收集）
3. **Phase 3: 完善与加固**（测试计划 → 基线测试 → 完整实现 → 完整测试 → 文档更新）
4. **Phase 4: 审查与发布**（代码审查 → 发布准备 → 部署上线 → 监控反馈）

**本项目应用**：
- 阶段 0-2：使用 Phase 1（探索与决策）
- 阶段 3：推荐使用 Phase 2（快速原型）+ Phase 3（完善加固）
- 阶段 4-5：使用完整的 VibCoding 循环
