# VibCoding 开发流程指南

## 文档说明

本文档定义了 Prompt 基座项目采用的核心开发流程：**VibCoding（振动式编码）**。

这是一种强调快速反馈、小步迭代、持续调整的开发方法论，适用于本项目的所有阶段。

---

## 什么是 VibCoding？

**VibCoding（Vibration-based Coding，振动式编码）** 是一种迭代开发思维：

- **核心理念**：快速反馈、小步迭代、持续调整
- **关键特点**：早期原型、快速验证、灵活调整
- **适用场景**：需求探索、架构重构、功能开发

## VibCoding 完整流程

```
┌─────────────────────────────────────────────────┐
│  Phase 1: 探索与决策（Explore & Decide）        │
└─────────────────────────────────────────────────┘
1. 需求探索（Requirement Exploration）
   ↓
2. 架构决策（ADR）
   ↓
3. 需求细化（Requirement Specification）
   ↓
   ┌──────────────────────────────────────────┐
   │  进入 VibCoding 循环                      │
   └──────────────────────────────────────────┘
   
┌─────────────────────────────────────────────────┐
│  Phase 2: 快速原型（Quick Prototype）           │
│  目标：验证可行性，获得早期反馈                  │
└─────────────────────────────────────────────────┘
4. 最小可行实现（MVP Implementation）
   - 只实现核心功能
   - 跳过边界情况
   - 快速验证想法
   ↓
5. 快速测试（Quick Test）
   - 手动测试核心路径
   - 验证基本功能
   ↓
6. 反馈收集（Feedback）
   - 是否符合预期？
   - 有什么问题？
   - 需要调整什么？
   ↓
   ┌──────────────────────────────────────────┐
   │  决策点：继续 or 调整 or 放弃？           │
   └──────────────────────────────────────────┘
   │
   ├─ 如果需要调整 → 回到步骤 4（重新实现）
   ├─ 如果不可行 → 回到步骤 2（重新决策）
   └─ 如果可行 → 继续下一阶段
   
┌─────────────────────────────────────────────────┐
│  Phase 3: 完善与加固（Refine & Solidify）       │
│  目标：补充细节，提高质量                        │
└─────────────────────────────────────────────────┘
7. 测试计划（Test Plan）
   - 基于原型经验制定
   - 覆盖发现的边界情况
   ↓
8. 测试现有功能（Baseline Testing）
   - 确保没有破坏现有功能
   ↓
9. 完善实现（Full Implementation）
   - 处理边界情况
   - 添加错误处理
   - 优化性能
   ↓
10. 完整测试（Full Testing）
    - 单元测试
    - 集成测试
    - 回归测试
    ↓
11. 文档更新（Documentation）
    - 基于实际实现更新文档
    ↓
    ┌──────────────────────────────────────────┐
    │  决策点：质量是否达标？                   │
    └──────────────────────────────────────────┘
    │
    ├─ 如果有问题 → 回到步骤 9（继续完善）
    └─ 如果达标 → 继续下一阶段
    
┌─────────────────────────────────────────────────┐
│  Phase 4: 审查与发布（Review & Release）        │
└─────────────────────────────────────────────────┘
12. 代码审查（Code Review）
    ↓
13. 发布准备（Release Preparation）
    ↓
14. 部署上线（Deployment）
    ↓
15. 监控反馈（Monitoring）
    ↓
    ┌──────────────────────────────────────────┐
    │  进入下一个迭代周期                       │
    └──────────────────────────────────────────┘
```

## VibCoding vs 传统瀑布式

| 维度 | 传统瀑布式 | VibCoding 振动式 |
|------|-----------|-----------------|
| **节奏** | 线性顺序，一次性完成 | 循环迭代，多次调整 |
| **反馈** | 最后才能看到结果 | 早期就能看到原型 |
| **风险** | 后期发现问题成本高 | 早期发现问题成本低 |
| **灵活性** | 难以调整方向 | 容易调整方向 |
| **文档** | 先写文档再写代码 | 先写原型再完善文档 |
| **测试** | 实现后才测试 | 边实现边测试 |
| **适用场景** | 需求明确、风险低 | 需求模糊、探索性强 |
| **心理压力** | 大（一次性完成）| 小（小步迭代）|

## VibCoding 的核心优势

### 1. 降低风险
- **早期发现问题**：原型阶段就能发现设计缺陷
- **容易回滚**：每次迭代都很小，回滚成本低
- **持续验证**：每次迭代都有反馈，不会走太远

### 2. 提高效率
- **快速反馈**：不用等到最后才知道结果
- **减少返工**：早期调整比后期修复成本低
- **并行工作**：可以同时进行多个小迭代

### 3. 降低心理压力
- **小步前进**：每次只做一小步，不会overwhelm
- **持续成就感**：每次迭代都有可见成果
- **容易坚持**：小目标更容易达成

### 4. 提高质量
- **持续测试**：边实现边测试，质量更高
- **持续改进**：每次迭代都在优化
- **经验积累**：每次迭代都学到新东西

## 实施建议

### 何时使用 VibCoding？

**✅ 推荐使用**：
- 需求不明确，需要探索
- 技术方案不确定，需要验证
- 风险较高，需要降低风险
- 时间紧迫，需要快速反馈

**⚠️ 谨慎使用**：
- 需求非常明确，不需要探索
- 技术方案成熟，不需要验证
- 风险很低，不需要降低风险
- 时间充裕，可以一次性完成

**❌ 不推荐使用**：
- 简单的配置修改
- 纯文档工作
- 已经验证过的重复工作

### 如何实施 VibCoding？

#### 步骤 1：定义 MVP
- 只包含核心功能
- 跳过边界情况
- 跳过错误处理
- 跳过性能优化

#### 步骤 2：快速实现
- 用最简单的方式实现
- 不追求完美
- 不过度设计
- 快速完成

#### 步骤 3：快速测试
- 手动测试核心路径
- 验证基本功能
- 记录发现的问题
- 不追求完整覆盖

#### 步骤 4：收集反馈
- 是否符合预期？
- 有什么问题？
- 需要调整什么？
- 是否继续？

#### 步骤 5：决策
- **继续**：进入下一次迭代
- **调整**：回到步骤 2，重新实现
- **放弃**：回到步骤 1，重新定义 MVP

#### 步骤 6：完善
- 处理边界情况
- 添加错误处理
- 优化性能
- 完善文档

## 常见问题

### Q1：VibCoding 会不会导致代码质量低？

**A**：不会。VibCoding 强调的是**迭代**，不是**草率**。

- Phase 2（快速原型）：可以质量低，目的是验证
- Phase 3（完善加固）：必须质量高，目的是交付
- Phase 4（审查发布）：严格质量控制

### Q2：VibCoding 会不会浪费时间？

**A**：不会。VibCoding 反而**节省时间**。

- 早期发现问题，避免后期返工
- 快速验证想法，避免走弯路
- 小步迭代，避免一次性失败

### Q3：VibCoding 适合所有项目吗？

**A**：不一定。VibCoding 适合**探索性强**的项目。

- ✅ 适合：新功能开发、架构重构、技术探索
- ⚠️ 谨慎：成熟功能优化、简单配置修改
- ❌ 不适合：纯文档工作、重复性工作

### Q4：如何判断 MVP 是否足够小？

**A**：MVP 应该满足以下条件：

- 实现时间 < 30分钟
- 测试时间 < 10分钟
- 只包含1-2个核心功能
- 可以独立验证

### Q5：如何判断是否应该继续迭代？

**A**：根据反馈决策：

- ✅ 继续：基本功能正常，符合预期
- 🔄 调整：有问题但可以修复
- ❌ 放弃：根本不可行，需要重新设计

## 本项目的经验教训

### 我犯的错误

**错误**：阶段 0-2 时，我跳过了需求细化和测试计划，直接写代码。

**后果**：
- 没有明确的验收标准
- 无法系统化测试
- 发现问题后才意识到缺少文档

**教训**：
- ✅ Phase 1（探索与决策）不能跳过任何步骤
- ✅ 需求细化和测试计划是必需的
- ✅ 即使用 VibCoding，也要完整执行 Phase 1

### 正确的做法

**阶段 0-2**：应该完整执行 Phase 1
```
1. 需求探索 ✅
2. 架构决策 ✅
3. 需求细化 ✅（补充）
4. 测试计划 ✅（补充）
5. 基线测试 ✅（补充）
```

**阶段 3**：应该用 VibCoding 循环
```
迭代 1：只迁移 ai_dev.md → 测试 → 反馈
迭代 2：迁移 ai-dev-script.py → 测试 → 反馈
迭代 3：迁移 ai-dev-config.json → 测试 → 反馈
Phase 3：完善与加固 → 完整测试 → 文档更新
```
