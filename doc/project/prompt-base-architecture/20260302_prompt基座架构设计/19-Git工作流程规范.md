# Git 工作流程规范

## 文档说明
本文档定义了 Prompt 基座项目的 Git 工作流程规范，确保每次修改都有清晰的提交记录。

## 核心原则

1. **小步提交**：每完成一个独立功能或修改，立即提交
2. **清晰描述**：提交信息要准确描述修改内容
3. **及时推送**：提交后立即推送到远程仓库
4. **文档同步**：代码和文档修改要一起提交

---

## 标准工作流程

### 步骤 1：检查当前状态
```bash
git status
```
**目的**：查看哪些文件被修改、新增或删除

### 步骤 2：添加文件到暂存区
```bash
git add -A
```
**说明**：
- `-A` 表示添加所有修改（包括新增、修改、删除）
- 也可以用 `git add .` 或指定具体文件

### 步骤 3：再次检查状态
```bash
git status
```
**目的**：确认哪些文件将被提交

### 步骤 4：提交更改
```bash
git commit -m "提交信息"
```

**提交信息格式**：
- 第一行：简短描述（50字以内）
- 空行
- 详细说明（可选，每行不超过72字）

**示例**：
```bash
git commit -m "完成阶段3：文件迁移到模块化目录结构

- 使用VibCoding方式执行（3次迭代+Phase3）
- 迁移ai-dev模块文件（steering/scripts/configs）
- 迁移doc-save模块文件（steering）
- 修复脚本配置文件路径，支持模块化目录
- 安装必要依赖（pyyaml, requests, python-dotenv）
- 创建requirements.txt
- 更新.gitignore（排除doc/personal/）
- 所有测试通过，功能正常"
```

### 步骤 5：推送到远程仓库
```bash
git push
```
**目的**：将本地提交同步到 GitHub

### 步骤 6：验证推送结果
```bash
git log --oneline -5
```
**目的**：查看最近5次提交记录，确认提交成功

---

## 提交信息规范

### 类型前缀（可选但推荐）
- `feat:` - 新功能
- `fix:` - 修复问题
- `docs:` - 文档更新
- `refactor:` - 代码重构
- `test:` - 测试相关
- `chore:` - 构建/工具相关

### 示例
```bash
# 新功能
git commit -m "feat: 添加模块版本切换功能"

# 修复问题
git commit -m "fix: 修复脚本配置文件路径错误"

# 文档更新
git commit -m "docs: 更新README添加Token验证指令"

# 代码重构
git commit -m "refactor: 重构模块加载器，支持依赖检查"
```

## 何时提交

### ✅ 应该提交的时机
1. 完成一个独立功能
2. 修复一个问题
3. 完成一次重构
4. 更新文档
5. 完成一个阶段的工作

### ❌ 不应该提交的时机
1. 代码有语法错误
2. 测试未通过
3. 功能未完成（除非是 WIP 提交）
4. 包含敏感信息（Token、密码等）

## 特殊情况处理

### 1. 忘记提交某个文件
```bash
# 添加遗漏的文件
git add 遗漏的文件

# 修改上一次提交（不创建新提交）
git commit --amend --no-edit

# 强制推送（谨慎使用）
git push --force
```

### 2. 提交信息写错
```bash
# 修改上一次提交信息
git commit --amend -m "新的提交信息"

# 强制推送
git push --force
```

### 3. 误提交敏感信息
```bash
# 立即从暂存区移除
git reset HEAD 敏感文件

# 如果已经提交，需要从历史中删除
# 建议：重新生成敏感信息（如Token），不要尝试修改历史
```

## 在新对话中的应用

### Kiro 应该做的
1. **每完成一个独立修改后**：
   - 执行 `git add -A`
   - 执行 `git status` 确认
   - 执行 `git commit -m "清晰的提交信息"`
   - 执行 `git push`
   - 执行 `git log --oneline -5` 验证

2. **提交信息要包含**：
   - 简短的标题（描述做了什么）
   - 详细的说明（列出具体修改项）

3. **提交时机**：
   - 完成文件迁移 → 提交
   - 完成代码修改 → 提交
   - 完成文档更新 → 提交
   - 完成测试验证 → 提交

### 用户的期望
- 每次对话结束时，所有修改都已提交并推送
- 提交记录清晰，方便回溯
- 不需要用户手动执行 Git 命令

## 检查清单

在结束对话前，确认：
- [ ] 所有修改已添加到暂存区（`git add -A`）
- [ ] 提交信息清晰准确（`git commit -m "..."`）
- [ ] 已推送到远程仓库（`git push`）
- [ ] 验证提交成功（`git log --oneline -5`）
- [ ] 工作目录干净（`git status` 显示 "nothing to commit"）
