# 双轨工作流选择器（完整版）

## 🎯 规则说明

本规则负责：
1. 判断是否可以快速跳过完整流程
2. 自动识别工作流类型（Prompt 工作流 / 代码工作流）
3. 执行对应的工作流（Phase 1-5）
4. 强制执行总结与反思（Phase 5）
5. 生成用户友好的使用指南文档 ⭐

**核心理念**：
- 简单修改快速跳过
- 复杂功能完整流程
- 每个项目都有总结反思
- 每个功能都有用户友好的使用指南

**重要说明**：
- 本规则是 Prompt 基座项目的一个核心功能模块
- 每次需求完成后生成的文档，是针对该需求功能的说明文档
- 不是针对整个系统的文档，而是针对当前开发的功能模块的文档

---

## 🔔 触发条件

当用户提出以下类型的需求时，自动触发：
- 开发新功能
- 修改现有功能
- 创建新文件
- 修改配置

**排除场景**：
- 纯粹的问答（如"什么是 VibCoding？"）
- 查看文件内容（如"查看 README.md"）
- 简单的命令执行（如"运行测试"）

## 📋 执行流程

### 步骤 1：快速跳过判断（优先级最高）

#### 1.1 检查是否启用快速跳过
从 `.kiro/kiro.yaml` 读取 `workflow.quick_skip.enabled`：
- 如果是 `false` → 跳过此步骤，进入步骤 2
- 如果是 `true` → 继续判断

#### 1.2 检查手动跳过关键词
检查用户输入是否包含手动跳过关键词：
- 中文："快速修改"、"简单修改"、"跳过流程"
- 英文："quick fix"、"simple change"、"skip workflow"

如果包含 → 询问用户确认：
```
🤔 检测到快速修改关键词

这是一个简单修改吗？可以跳过完整的工作流吗？

适合跳过的情况：
- 简单配置修改（< 10 行）
- 纯文档错别字修正
- 简单的规则微调（不改变核心逻辑）
- 格式化、注释修改

请确认：
A. 是，跳过流程（直接执行修改，5-10 分钟）
B. 否，执行完整流程（Phase 1-5，2.5-8 小时）

请选择：A / B
```

如果用户选择 A：
```
✅ 已跳过完整工作流

执行模式：快速修改模式
- 不需要 Phase 1-4
- 直接执行修改
- 修改完成后简单验证
- 不需要 Phase 5（总结反思）

开始执行修改...
```

如果用户选择 B：
```
✅ 执行完整工作流

继续步骤 2：工作流类型识别
```

#### 1.3 检查自动跳过条件
如果用户没有使用手动跳过关键词，检查自动跳过条件：

**条件 1：修改行数 < 10 行**
```
分析用户需求中提到的修改内容：
- 如果明确说明修改行数 < 10 行 → 询问是否跳过
- 如果无法判断 → 继续检查其他条件
```

**条件 2：简单配置文件**
```
分析用户需求中提到的文件类型：
- 如果是 *.yaml、*.json、*.txt → 询问是否跳过
- 如果是其他类型 → 继续检查其他条件
```

**条件 3：错别字、格式化、注释修改**
```
分析用户需求中的关键词：
- 如果包含"错别字"、"格式化"、"注释"、"typo"、"format" → 询问是否跳过
- 如果不包含 → 进入步骤 2
```

**询问方式**：
```
🤔 检测到可能是简单修改

根据你的需求描述，这可能是一个简单修改：
- [检测到的条件，如"修改行数 < 10 行"]

是否跳过完整的工作流？

请确认：
A. 是，跳过流程（直接执行修改，5-10 分钟）
B. 否，执行完整流程（Phase 1-5，2.5-8 小时）

请选择：A / B
```

### 步骤 2：工作流类型识别

#### 2.1 检查手动覆盖
检查用户输入是否包含手动覆盖关键词：
- "使用 Prompt 工作流"、"use prompt workflow"
- "使用代码工作流"、"use code workflow"

如果包含 → 使用用户指定的工作流：
```
✅ 已识别手动覆盖

用户指定：[Prompt 工作流 / 代码工作流]

开始执行 Phase 1：探索与决策
```

#### 2.2 分析需求中的文件路径
从用户需求中提取文件路径，匹配 `workflow.auto_switch` 规则：

**Prompt 工作流规则**：
- `.kiro/steering/*.md`
- `.kiro/hooks/*.json`
- `doc/project/*.md`
- `.kiro/kiro.yaml`
- `.kiro/templates/*.md`

**代码工作流规则**：
- `**/*.py`
- `.kiro/modules/*/scripts/*.py`
- `scripts/*.py`
- `tests/*.py`

**匹配示例**：
```
示例 1：
用户："修改 .kiro/steering/git_rollback.md"
→ 匹配规则：.kiro/steering/*.md
→ 工作流：Prompt 工作流

示例 2：
用户："开发 scripts/data_processor.py"
→ 匹配规则：scripts/*.py
→ 工作流：代码工作流
```

如果匹配成功：
```
✅ 已自动识别工作流类型

文件路径：[用户提到的文件路径]
匹配规则：[匹配的规则]
工作流类型：[Prompt 工作流 / 代码工作流]

开始执行 Phase 1：探索与决策
```

#### 2.3 分析需求中的关键词
如果无法从文件路径识别，分析关键词：

**Prompt 工作流关键词**：
- "Steering 规则"、"Hook 规则"、"配置文件"、"文档"
- "规则"、"流程"、"指南"、"steering"、"hook"

**代码工作流关键词**：
- "Python 代码"、"脚本"、"模块"、"功能"、"函数"、"类"
- "测试"、"API"、"工具"、"script"、"module"、"function"

如果匹配成功：
```
✅ 已根据关键词识别工作流类型

关键词：[识别到的关键词]
工作流类型：[Prompt 工作流 / 代码工作流]

开始执行 Phase 1：探索与决策
```

#### 2.4 使用默认工作流
如果仍然无法判断，使用默认工作流（从 `workflow.default` 读取）：

```
⚠️ 无法自动识别工作流类型

使用默认工作流：[Prompt 工作流 / 代码工作流]

如果需要切换工作流，请明确说明：
- "使用 Prompt 工作流"
- "使用代码工作流"

开始执行 Phase 1：探索与决策
```

### 步骤 4：强制执行 Phase 5（总结与反思）

#### 4.1 触发时机

**时机 1：项目完成时（必须执行）**
Phase 4 完成后，自动触发 Phase 5：

```
✅ Phase 4 已完成

接下来进入 Phase 5：总结与反思

这是一个强制步骤，用于：
- 识别项目中遇到的问题
- 分析根本原因
- 提取经验教训
- 制定改进措施
- 沉淀知识

预计时间：30-60 分钟

开始执行 Phase 5...
```

**时机 2：遇到重大问题时（可选执行）**
在 Phase 2-4 过程中，如果遇到以下情况，询问是否提前执行 Phase 5：
- 需求理解错误
- 架构设计错误
- 实现方向错误
- 需要回滚重做

```
⚠️ 检测到重大问题

问题类型：[需求理解错误 / 架构设计错误 / ...]
问题描述：[简要描述]

建议提前执行 Phase 5：总结与反思

这样可以：
- 及时分析问题原因
- 避免继续走错方向
- 为后续调整提供指导

```

```
用户提出需求
    ↓
【步骤 1】快速跳过判断
    ├─ 检测到手动跳过关键词？
    │   ├─ 是 → 询问用户确认
    │   │       ├─ 用户确认跳过 → 快速修改模式（结束）
    │   │       └─ 用户拒绝跳过 → 步骤 2
    │   └─ 否 → 检查自动跳过条件
    │           ├─ 满足条件 → 询问用户确认
    │           │             ├─ 用户确认跳过 → 快速修改模式（结束）
    │           │             └─ 用户拒绝跳过 → 步骤 2
    │           └─ 不满足 → 步骤 2
    ↓
【步骤 2】工作流类型识别
    ├─ 检查手动覆盖？
    │   ├─ 是 → 使用用户指定的工作流
    │   └─ 否 → 分析文件路径
    │           ├─ 匹配成功 → 使用匹配的工作流
    │           └─ 匹配失败 → 分析关键词
    │                       ├─ 匹配成功 → 使用匹配的工作流
    │                       └─ 匹配失败 → 使用默认工作流
    ↓
【步骤 3】执行对应工作流
    ├─ Prompt 工作流 → Phase 1-4
    └─ 代码工作流 → Phase 1-4
    ↓
【步骤 4】强制执行 Phase 5
    └─ 总结与反思（5 个强制项）
```

---

## 📚 规则总结

### 核心优势

1. **智能识别**：
   - ✅ 按文件路径自动识别
   - ✅ 按关键词自动识别
   - ✅ 支持手动覆盖

2. **快速跳过**：
   - ✅ 手动跳过关键词
   - ✅ 自动跳过条件
   - ✅ 用户确认机制

3. **强制总结反思**：
   - ✅ 每个项目都有 Phase 5
   - ✅ 5 个强制项不可跳过
   - ✅ 形成知识积累

4. **防止需求理解错误**：
   - ✅ Phase 1 的 5 个强制检查项
   - ✅ Agent 主动提问机制
   - ✅ 参考案例（正确 vs 错误）

5. **用户友好文档** ⭐：
   - ✅ 每个项目都生成使用指南
   - ✅ 人性化的写作风格
   - ✅ 新用户快速上手
   - ✅ 统一的文档结构

### 文档输出

每个项目完成后，会生成两份文档：

**文档 1：总结与反思文档**
- 文件名：`YYYYMMDD_项目名-总结与反思.md`
- 目标读者：团队成员
- 内容：问题分析、根本原因、经验教训、改进措施
- 用途：团队复盘、知识积累

**文档 2：用户友好总结文档** ⭐
- 文件名：`YYYYMMDD_项目名-使用指南.md`
- 目标读者：新用户
- 内容：核心特性、快速开始、常见问题、配置说明
- 用途：新用户快速上手、功能推广
- 风格：人性化、易读、实用

### 决策优先级

```
1. 快速跳过判断（最高优先级）
   ├─ 手动跳过关键词
   └─ 自动跳过条件

2. 手动覆盖（高优先级）
   └─ 用户明确指定工作流类型

3. 自动识别（中优先级）
   ├─ 文件路径匹配
   └─ 关键词匹配

4. 默认工作流（最低优先级）
   └─ 使用 workflow.default 配置

5. Phase 5 总结反思（强制执行）
   ├─ 项目完成后必须执行
   └─ 生成两份文档（总结反思 + 使用指南）
```

## 🎯 执行约束（强制遵守）

### 约束 1：步骤反馈
每一步骤完成后，必须向用户反馈：

```
✅ [步骤名称] 已完成

确认项：
- [已确认的信息 1]
- [已确认的信息 2]

待确认项：
- [需要提问的问题 1]（如果没有，写"无"）
```

### 约束 2：提问具体化
所有提问必须清晰、具体，避免模糊：

❌ 错误："你确认一下"
✅ 正确："请确认目标用户是 Agent 还是人类用户？"

### 约束 3：文档同步
所有确认的信息必须同步记录到对应文档。

### 约束 4：Phase 5 不可跳过
除非是快速修改模式，否则 Phase 5 必须执行。
