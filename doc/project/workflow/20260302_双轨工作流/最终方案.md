## 📋 优化后的 kiro.yaml 配置

```yaml
# Kiro 全局配置文件

version: "1.0"

# 工作流配置（优化版）
workflow:
  # 默认工作流
  default: "prompt"  # prompt / code
  
  # 按目录/文件自动切换工作流
  auto_switch:
    # Prompt 工作流规则
    - pattern: ".kiro/steering/*.md"
      workflow: "prompt"
      description: "Steering 规则文件"
    
    - pattern: ".kiro/hooks/*.json"
      workflow: "prompt"
      description: "Hook 规则文件"
    
    - pattern: "doc/project/*.md"
      workflow: "prompt"
      description: "项目文档"
    
    - pattern: ".kiro/kiro.yaml"
      workflow: "prompt"
      description: "配置文件"
    
    # 代码工作流规则
    - pattern: "**/*.py"
      workflow: "code"
      description: "Python 代码文件"
    
    - pattern: ".kiro/modules/*/scripts/*.py"
      workflow: "code"
      description: "模块脚本文件"
    
    - pattern: "scripts/*.py"
      workflow: "code"
      description: "脚本文件"
  
  # 快速跳过配置
  quick_skip:
    enabled: true  # 是否启用快速跳过
    
    # 自动跳过条件（满足任一条件即可跳过）
    auto_skip_conditions:
      - type: "line_count"
        threshold: 10
        description: "修改行数 < 10 行"
      
      - type: "file_type"
        patterns: ["*.yaml", "*.json", "*.txt"]
        description: "简单配置文件"
      
      - type: "change_type"
        types: ["typo", "format", "comment"]
        description: "错别字、格式化、注释修改"
    
    # 手动跳过关键词
    manual_skip_keywords:
      - "快速修改"
      - "简单修改"
      - "跳过流程"
      - "quick fix"
      - "simple change"

# 当前激活的模块版本
active_modules:
  ai-dev: "v1.0"
  doc-save: "v1.0"
  prompt-manager: "v1.0"

# 实验性功能开关
experimental:
  git_auto_rollback: true
  vibcoding_strict_mode: true

# Git 回滚配置
rollback:
  backup:
    enabled: true
    push_to_remote: false
    auto_delete_after_days: 7
```

# 工作流选择器（增强版）

## 🎯 规则说明

本规则负责：
1. 自动识别工作流类型（Prompt 工作流 / 代码工作流）
2. 判断是否可以快速跳过完整流程
3. 提供灵活的手动覆盖机制

---

## 🔔 触发条件

当用户提出以下类型的需求时，自动触发：
- 开发新功能
- 修改现有功能
- 创建新文件
- 修改配置

## 📋 执行流程

### 步骤 1：快速跳过判断（优先级最高）

#### 1.1 检查是否启用快速跳过
从 `.kiro/kiro.yaml` 读取 `workflow.quick_skip.enabled`：
- 如果是 `false` → 跳过此步骤，进入步骤 2
- 如果是 `true` → 继续判断

#### 1.2 检查手动跳过关键词
检查用户输入是否包含手动跳过关键词：
- "快速修改"、"简单修改"、"跳过流程"
- "quick fix"、"simple change"

如果包含 → 询问用户确认：
```
🤔 检测到快速修改关键词

这是一个简单修改吗？可以跳过完整的 VibCoding 流程吗？

如果是以下情况，建议跳过：
- 简单配置修改（< 10 行）
- 纯文档错别字修正
- 简单的规则微调（不改变核心逻辑）
- 格式化、注释修改

请确认：
A. 是，跳过流程（直接执行修改）
B. 否，执行完整流程（Phase 1-4）

请选择：A / B
```

如果用户选择 A：
```
✅ 已跳过 VibCoding 流程

执行模式：快速修改模式
- 不需要 Phase 1-4
- 直接执行修改
- 修改完成后简单验证

开始执行修改...
```

如果用户选择 B：
```
✅ 执行完整 VibCoding 流程

继续步骤 2：工作流类型识别
```

#### 1.3 检查自动跳过条件
如果用户没有使用手动跳过关键词，检查自动跳过条件：

**条件 1：修改行数 < 10 行**
```
分析用户需求中提到的修改内容：
- 如果明确说明修改行数 < 10 行 → 询问是否跳过
- 如果无法判断 → 继续检查其他条件
```

**条件 2：简单配置文件**
```
分析用户需求中提到的文件类型：
- 如果是 *.yaml、*.json、*.txt → 询问是否跳过
- 如果是其他类型 → 继续检查其他条件
```

**条件 3：错别字、格式化、注释修改**
```
分析用户需求中的关键词：
- 如果包含"错别字"、"格式化"、"注释" → 询问是否跳过
- 如果不包含 → 进入步骤 2
```

**询问方式**：
```
🤔 检测到可能是简单修改

根据你的需求描述，这可能是一个简单修改：
- [检测到的条件，如"修改行数 < 10 行"]

是否跳过完整的 VibCoding 流程？

请确认：
A. 是，跳过流程（直接执行修改）
B. 否，执行完整流程（Phase 1-4）

请选择：A / B
```

---

### 步骤 2：工作流类型识别

#### 2.1 分析需求中的文件路径
从用户需求中提取文件路径，匹配 `workflow.auto_switch` 规则：

```
示例 1：
用户："修改 .kiro/steering/git_rollback.md"
→ 匹配规则：.kiro/steering/*.md
→ 工作流：prompt

示例 2：
用户："开发 scripts/data_processor.py"
→ 匹配规则：scripts/*.py
→ 工作流：code

示例 3：
用户："修改 doc/project/需求文档.md"
→ 匹配规则：doc/project/*.md
→ 工作流：prompt
```

#### 2.2 如果匹配成功
```
✅ 已自动识别工作流类型

文件路径：[用户提到的文件路径]
匹配规则：[匹配的规则]
工作流类型：[Prompt 工作流 / 代码工作流]

开始执行 Phase 1：探索与决策
```

#### 2.3 如果匹配失败（无法从需求中提取文件路径）
分析需求中的关键词：

**Prompt 工作流关键词**：
- "Steering 规则"、"Hook 规则"、"配置文件"、"文档"
- "规则"、"流程"、"指南"

**代码工作流关键词**：
- "Python 代码"、"脚本"、"模块"、"功能"
- "测试"、"API"、"工具"

如果匹配成功：
```
✅ 已根据关键词识别工作流类型

关键词：[识别到的关键词]
工作流类型：[Prompt 工作流 / 代码工作流]

开始执行 Phase 1：探索与决策
```

#### 2.4 如果仍然无法判断
使用默认工作流（从 `workflow.default` 读取）：

```
⚠️ 无法自动识别工作流类型

使用默认工作流：[Prompt 工作流 / 代码工作流]

如果需要切换工作流，请明确说明：
- "使用 Prompt 工作流"
- "使用代码工作流"

开始执行 Phase 1：探索与决策
```

---

### 步骤 3：手动覆盖机制

用户可以在任何时候明确指定工作流类型：

```
用户："使用 Prompt 工作流，开发 Git 回滚功能"
→ 强制使用 Prompt 工作流

用户："使用代码工作流，开发项目管理工具"
→ 强制使用代码工作流
```

手动覆盖优先级最高，忽略自动识别结果。

## 📊 决策流程图

```
用户提出需求
    ↓
步骤 1：快速跳过判断
    ↓
检查手动跳过关键词？
    ├─ 是 → 询问用户确认
    │       ├─ 用户确认跳过 → 快速修改模式（结束）
    │       └─ 用户拒绝跳过 → 步骤 2
    └─ 否 → 检查自动跳过条件
            ├─ 满足条件 → 询问用户确认
            │             ├─ 用户确认跳过 → 快速修改模式（结束）
            │             └─ 用户拒绝跳过 → 步骤 2
            └─ 不满足 → 步骤 2
    ↓
步骤 2：工作流类型识别
    ↓
检查手动覆盖？
    ├─ 是 → 使用用户指定的工作流
    └─ 否 → 分析文件路径
            ├─ 匹配成功 → 使用匹配的工作流
            └─ 匹配失败 → 分析关键词
                        ├─ 匹配成功 → 使用匹配的工作流
                        └─ 匹配失败 → 使用默认工作流
    ↓
步骤 3：加载对应工作流
    ↓
执行 Phase 1-4
```
