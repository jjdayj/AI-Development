# Git 工作流程自动化规则

## 核心原则
在每次完成独立修改后，自动执行 Git 提交和推送流程。

## 执行时机

### ✅ 应该立即提交的情况
1. 完成一个独立功能的开发
2. 完成一次代码修复
3. 完成文档更新
4. 完成一个阶段的工作
5. 完成文件迁移或重构

### 标准流程

每次修改完成后，按以下顺序执行：

```bash
# 1. 检查当前状态
git status

# 2. 添加所有修改
git add -A

# 3. 再次确认
git status

# 4. 提交（使用清晰的提交信息）
git commit -m "简短描述

- 详细修改项1
- 详细修改项2
- 详细修改项3"

# 5. 推送到远程
git push

# 6. 验证提交成功
git log --oneline -5
```

## 提交信息规范

### 格式
- 第一行：简短描述（50字以内）
- 空行
- 详细说明（列表形式，每项一行）

### 类型前缀（推荐）
- `feat:` - 新功能
- `fix:` - 修复问题
- `docs:` - 文档更新
- `refactor:` - 代码重构
- `test:` - 测试相关
- `chore:` - 构建/工具相关

### 示例
```
feat: 添加模块版本切换功能

- 实现版本切换逻辑
- 添加版本验证
- 更新文档说明
```

## 注意事项

1. **不要提交敏感信息**：
   - .env 文件已在 .gitignore 中
   - Token、密码等不应出现在代码中

2. **确保代码可运行**：
   - 提交前确保没有语法错误
   - 关键功能要测试通过

3. **提交粒度**：
   - 每次提交应该是一个独立的功能点
   - 避免一次提交包含多个不相关的修改

## 在对话中的应用

作为 Kiro，我应该：
1. 在完成每个独立修改后，主动执行 Git 提交流程
2. 提供清晰的提交信息，说明做了什么修改
3. 在对话结束前，确保所有修改都已提交并推送
4. 如果用户询问修改内容，可以通过 `git log` 展示提交记录
