# 双轨工作流选择器（简化版）

## 🎯 规则说明

本规则负责：
1. 自动识别工作流类型（Prompt 工作流 / 代码工作流）
2. 执行对应的工作流（Phase 1-4）

**核心理念**：
- 快速识别工作流类型
- 执行核心开发流程
- 不包含 Phase 5 总结反思
- 不生成文档

**适用场景**：
- 快速开发迭代
- 不需要详细文档的项目
- 个人项目或实验性项目

---

## 🔔 触发条件

### 关键词识别
用户输入中包含以下任一关键词时触发：
- 开发新功能、实现功能、添加功能
- 创建模块、新建模块、设计模块
- 功能设计、架构设计
- 重构、优化架构

### 排除场景
- 纯粹的问答（如"什么是 VibCoding？"）
- 查看文件内容（如"查看 README.md"）
- 简单的命令执行（如"运行测试"）

---

## 📋 执行流程

### 步骤 1：工作流类型识别

#### 1.1 检查手动覆盖
检查用户输入是否包含手动覆盖关键词：
- "使用 Prompt 工作流"、"use prompt workflow"
- "使用代码工作流"、"use code workflow"

如果包含 → 使用用户指定的工作流：
```
✅ 已识别手动覆盖

用户指定：[Prompt 工作流 / 代码工作流]

开始执行 Phase 1：探索与决策
```

#### 1.2 分析需求中的文件路径
从用户需求中提取文件路径，匹配工作流规则：

**Prompt 工作流规则**：
- `.kiro/steering/*.md`
- `.kiro/hooks/*.json`
- `doc/project/*.md`
- `.kiro/config.yaml`
- `.kiro/templates/*.md`

**代码工作流规则**：
- `**/*.py`
- `.kiro/modules/*/scripts/*.py`
- `scripts/*.py`
- `tests/*.py`

**匹配示例**：
```
示例 1：
用户："修改 .kiro/steering/git_rollback.md"
→ 匹配规则：.kiro/steering/*.md
→ 工作流：Prompt 工作流

示例 2：
用户："开发 scripts/data_processor.py"
→ 匹配规则：scripts/*.py
→ 工作流：代码工作流
```

如果匹配成功：
```
✅ 已自动识别工作流类型

文件路径：[用户提到的文件路径]
匹配规则：[匹配的规则]
工作流类型：[Prompt 工作流 / 代码工作流]

开始执行 Phase 1：探索与决策
```

#### 1.3 分析需求中的关键词
如果无法从文件路径识别，分析关键词：

**Prompt 工作流关键词**：
- "Steering 规则"、"Hook 规则"、"配置文件"、"文档"
- "规则"、"流程"、"指南"、"steering"、"hook"

**代码工作流关键词**：
- "Python 代码"、"脚本"、"模块"、"功能"、"函数"、"类"
- "测试"、"API"、"工具"、"script"、"module"、"function"

如果匹配成功：
```
✅ 已根据关键词识别工作流类型

关键词：[识别到的关键词]
工作流类型：[Prompt 工作流 / 代码工作流]

开始执行 Phase 1：探索与决策
```

#### 1.4 使用默认工作流
如果仍然无法判断，使用默认工作流（从 `default_workflow` 读取）：

```
⚠️ 无法自动识别工作流类型

使用默认工作流：[Prompt 工作流 / 代码工作流]

如果需要切换工作流，请明确说明：
- "使用 Prompt 工作流"
- "使用代码工作流"

开始执行 Phase 1：探索与决策
```

---

### 步骤 2：执行对应工作流

#### 2.1 如果是 Prompt 工作流
执行 Prompt 工作流 Phase 1-4（轻量级版本）

#### 2.2 如果是代码工作流
执行代码工作流 Phase 1-4（完整版本）

---

## 📊 决策流程图

```
用户提出需求
    ↓
【步骤 1】工作流类型识别
    ├─ 检查手动覆盖？
    │   ├─ 是 → 使用用户指定的工作流
    │   └─ 否 → 分析文件路径
    │           ├─ 匹配成功 → 使用匹配的工作流
    │           └─ 匹配失败 → 分析关键词
    │                       ├─ 匹配成功 → 使用匹配的工作流
    │                       └─ 匹配失败 → 使用默认工作流
    ↓
【步骤 2】执行对应工作流
    ├─ Prompt 工作流 → Phase 1-4
    └─ 代码工作流 → Phase 1-4
    ↓
完成（无 Phase 5）
```

## 📚 规则总结

### 核心优势

1. **智能识别**：
   - ✅ 按文件路径自动识别
   - ✅ 按关键词自动识别
   - ✅ 支持手动覆盖

2. **简化流程**：
   - ✅ 只执行 Phase 1-4
   - ✅ 不包含 Phase 5 总结反思
   - ✅ 不生成文档
   - ✅ 快速开发迭代

### 决策优先级

```
1. 手动覆盖（高优先级）
   └─ 用户明确指定工作流类型

2. 自动识别（中优先级）
   ├─ 文件路径匹配
   └─ 关键词匹配

3. 默认工作流（最低优先级）
   └─ 使用 default_workflow 配置
```

## 🎯 执行约束（强制遵守）

### 约束 1：步骤反馈
每一步骤完成后，必须向用户反馈：

```
✅ [步骤名称] 已完成

确认项：
- [已确认的信息 1]
- [已确认的信息 2]

待确认项：
- [需要提问的问题 1]（如果没有，写"无"）
```

### 约束 2：提问具体化
所有提问必须清晰、具体，避免模糊：

❌ 错误："你确认一下"
✅ 正确："请确认目标用户是 Agent 还是人类用户？"

### 约束 3：文档同步
所有确认的信息必须同步记录到对应文档。
