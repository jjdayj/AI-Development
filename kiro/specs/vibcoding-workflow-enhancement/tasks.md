## 注意事项

### Prompt 工程实现特点

1. **修改 workflow_selector.md**：
   - 所有实现都在 `.kiro/modules/workflow/v1.0/steering/workflow_selector.md` 文件中
   - 使用 Markdown 格式编写 Prompt 逻辑
   - 使用清晰的章节结构组织钩子逻辑

2. **钩子实现方式**：
   - 前置钩子：在现有工作流开始前添加新章节
   - 后置钩子：在现有工作流结束后添加新章节
   - 中断处理：在工作流说明中添加切换命令

3. **测试方式**：
   - 由于是 Prompt 工程，测试主要通过手动验证
   - 创建测试配置文件和测试场景
   - 运行 Kiro AI 并观察行为是否符合预期

4. **可选任务标记**：
   - 标记为 `*` 的任务是可选的测试任务
   - 核心实现任务（1-5, 7）必须完成
   - 测试任务（1.1-1.4, 2.1-2.3, 3.1, 4.1, 5.1, 6）可以根据时间安排选择性完成

### 实现顺序

建议按以下顺序实现：
1. 任务 1：前置钩子（最重要，工作流入口）
2. 任务 2：后置钩子（工作流出口）
3. 任务 5：工作流集成点（确保钩子正确触发）
4. 任务 7：验证集成（确保与其他模块兼容）
5. 任务 3：需求切换处理（高级功能）
6. 任务 4：错误处理（增强健壮性）
7. 任务 6：测试配置（可选）
8. 任务 8：最终检查点

### 关键集成点

1. **与动态模块加载器集成**：
   - current_context.yaml 必须使用嵌套结构（active_requirement + environment + updated_at）
   - 字段名称必须一致

2. **与 Git 提交模块集成**：
   - git_commit_prefix 格式：`feat({module}-{version})`（保留 'v' 前缀）
   - 通过关键词触发：`提交代码: {描述}`
   - Git 提交模块自动从 current_context.yaml 读取信息

3. **与 VibCoding 工作流集成**：
   - 不修改现有 5 个核心阶段
   - 通过钩子方式增强
   - 提供降级选项
