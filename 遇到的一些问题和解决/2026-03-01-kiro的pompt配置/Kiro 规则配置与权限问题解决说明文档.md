# Kiro 规则配置与权限问题解决说明文档（最终修订版）

## 一、跨项目复用说明（核心）

### 快速复用至其他项目

只需将当前项目根目录下的 `.kiro` 文件夹**完整复制**到目标项目的根目录，即可让该项目自动继承所有规则（文档保存、对话存档、权限信任）；若需添加新的自定义规则，将新的 `steering` 规则文件放入目标项目的 `.kiro/steering/` 目录即可，无需重复配置。

## 二、背景与目标

在使用 Kiro 进行开发时，希望实现以下自动化流程：

1. 每次操作生成独立操作文档，按**步骤分组**的目录结构保存，确保操作文档与对应对话文件一一对应。
2. 对话历史自动拆分保存，文件开头包含需求总结，且与操作文档强绑定。
3. 全程无需手动干预，Kiro 自动执行文件读写、目录创建操作。

## 三、核心需求拆解

### （一）文档自动生成

1. 每步操作生成独立 `.md` 文档，按「会话根目录→步骤子目录」层级存放。
2. 目录不存在时自动创建，文件名遵循「步骤名称_时间戳.md」规范。
3. 新增**会话总览文档**，汇总所有步骤的文档索引，便于快速查找。

### （二）对话自动存档

1. 每个对话框（单次提问 + 回复）保存为独立 `.txt` 文件，与对应步骤的操作文档存放在同一子目录。
2. 文件开头总结用户提问核心需求 + 涉及的操作类型，标注关联的操作文档名称。
3. 对话文件命名与步骤强关联，实现 “一步一文档一对话” 的精准匹配。

### （三）权限自动授权

1. 避免每次创建目录、保存文件时手动确认授权。
2. 实现 “一次配置，永久生效”，覆盖 CMD 与 PowerShell 两种命令环境。

## 四、配置过程与关键问题

### 1. 规则文件位置与格式问题

- **初始尝试**：将规则放在 `prompt/` 目录下，通过发送长提示词加载，无法实现持久化生效。

- **官方方案**：Kiro 官方推荐使用 `.kiro/steering/` 目录，通过 `steering` 文件实现项目级持久化规则，无需每次重复发送提示词。

- 格式修正

  ：

  1. 规则文件必须以 `---` 包裹 `front-matter` 配置（如 `inclusion: always`），否则 Kiro 无法识别全局生效规则。
  2. 初始规则未明确 “步骤层级”，导致文档与对话文件杂乱存放，无法一一对应。

  

### 2. 规则未生效的排查

- **问题表现**：发送指令后未按预期创建层级目录，仅生成根目录下的零散文件。

- 核心原因

  ：

  1. `inclusion: always` 未包裹在 `---` 内，Kiro 未加载全局规则。
  2. 初始规则仅定义 “会话根目录”，未设计 “步骤子目录”，缺乏层级划分逻辑。
  3. 变量（如 `[对话框主题]` `[步骤序号]`）写法不明确，Kiro 执行时无统一标准。

  

- **解决方案**：修正 `front-matter` 格式，重新设计**按步骤分组**的目录结构与命名规则。

### 3. 重复授权问题

- **问题表现**：第一次授权 `if *` 后，第二次执行目录创建仍需授权。

- 核心原因

  ：

  1. Kiro 对不同命令类型的信任是**分开管理**的，无全局统一权限。
  2. 第一次授权的是 CMD 命令 `if not exist ... mkdir`，第二次执行的是 PowerShell 命令 `New-Item`，不在同一信任规则范围内。

  

- **解决方案**：对 CMD 与 PowerShell 的核心文件操作命令，分别进行 `Base` 级别授权，覆盖所有读写场景。

### 4. 文档与对话文件匹配问题（新增核心问题）

- **问题表现**：所有操作文档与对话文件混存于同一会话根目录，序号与步骤无关联，后续查找困难。
- **核心原因**：初始规则未建立 “步骤 - 文档 - 对话” 的强绑定关系，仅按会话维度存储，缺乏步骤层级划分。
- **解决方案**：在规则中新增**步骤子目录**设计，让每个步骤的操作文档与对话文件同目录存放，命名上共享步骤标识。

## 五、最终解决方案

### （一）规则文件重新配置（核心修订）

在项目根目录的 `.kiro/steering/` 下，替换为**按步骤分组**的 `doc_save_rules.md`，内容如下：

markdown











```
---
inclusion: always  # Kiro 官方全局默认生效配置，必须包裹在---内
---
# Kiro 执行默认规则（全局生效）
## 一、目录结构规范（核心修订：按步骤分组）
1. **会话根目录**：`doc/[会话主题]-[会话时间戳]/`
    - 会话主题：提取本次会话的核心目标（如“谷歌账号注册及Git配置”）。
    - 会话时间戳：`YYYYMMDD_HHMMSS`（会话开始时生成，全程唯一）。
2. **步骤子目录**：在会话根目录下，为每个操作步骤创建独立子目录。
    - 命名格式：`[步骤序号]-[步骤名称]-[步骤时间戳]/`。
    - 步骤序号：从 `01` 开始递增，与任务流程严格对应。
    - 步骤时间戳：`YYYYMMDD_HHMMSS`（该步骤开始执行时生成）。
3. **汇总目录**：无需单独创建，汇总文档直接存放在会话根目录。

## 二、文件命名与存放规范
### （一）操作文档（.md）
1. 存放位置：对应步骤子目录内。
2. 命名格式：`[步骤名称]_[步骤时间戳].md`。
3. 示例：`谷歌账号注册_20260301_144600.md`。

### （二）对话文件（.txt）
1. 存放位置：与对应操作文档**同一步骤子目录**内（核心：强绑定）。
2. 命名格式：`对话_[步骤序号]-[步骤名称]_[步骤时间戳].txt`。
3. 示例：`对话_01-谷歌账号注册_20260301_144600.txt`。
4. 序号规则：对话序号与步骤序号保持一致，无需额外递增。

### （三）会话总览文档（.md）
1. 存放位置：会话根目录下。
2. 命名格式：`会话总览_[会话时间戳].md`。
3. 作用：生成所有步骤的索引，快速定位文档与对话。

## 三、文档内容规范
### （一）操作文档（.md）
1. 开头必含：步骤目标、适用场景、核心操作清单。
2. 正文：可直接执行的步骤/命令，关键内容加粗标注。

### （二）对话文件（.txt）
1. 开头必含（核心：交叉引用）：
    - 核心总结：用户本次提问的核心需求+涉及的操作类型。
    - 关联文档：本次对话对应的操作文档名称（如「关联操作文档：谷歌账号注册_20260301_144600.md」）。
2. 正文：完整的用户提问与 Kiro 回复内容。

### （三）会话总览文档（.md）
1. 开头：会话核心目标、执行时间、总步骤数。
2. 正文：以列表形式生成索引，格式为「[步骤序号] [步骤名称] → 操作文档路径 → 对话文件路径」。

## 四、执行流程规范
1. **会话启动**：创建会话根目录，生成空的会话总览文档。
2. **步骤执行**：
    - 为当前步骤创建步骤子目录。
    - 生成该步骤的操作文档与对话文件，存入步骤子目录。
    - 更新会话总览文档，添加该步骤的索引条目。
3. **会话结束**：补全会话总览文档的执行总结，完成所有文件保存。
```

### （二）权限配置（完整覆盖）

在 Kiro 首次执行以下命令时，**全部选择「Base」级别授权**，实现永久自动执行：

表格







|    命令类型     | 需授权的命令前缀 |              适用场景               |
| :-------------: | :--------------: | :---------------------------------: |
|    CMD 命令     |      `if *`      |  CMD 环境下的目录存在性判断与创建   |
|    CMD 命令     |     `echo *`     |      CMD 环境下的文件内容写入       |
| PowerShell 命令 |   `New-Item *`   | PowerShell 环境下的目录 / 文件创建  |
| PowerShell 命令 |   `Out-File *`   |   PowerShell 环境下的文件内容写入   |
| PowerShell 命令 | `Set-Content *`  | PowerShell 环境下的文件内容覆盖写入 |

## 七、总结

### （一）核心优化成果

本次配置通过**三次核心优化**，彻底解决了文档混乱、权限重复授权、提示词冗余的问题：

1. **规则层面**：重新设计「会话根目录→步骤子目录」的层级结构，实现操作文档与对话文件的一一对应。
2. **权限层面**：全覆盖 CMD 与 PowerShell 的核心文件操作命令，实现 “一次授权，永久生效”。
3. **提示词层面**：删除所有保存相关指令，仅保留任务流程控制，让 `steering` 规则完全接管自动化保存逻辑。

最终实现了 “规则永久生效、文档结构清晰、操作全程自动化” 的目标，完美适配 vibcoding 分步执行的使用场景。

### （二）跨项目复用再次强调

1. 复用已有规则：将当前项目的 `.kiro` 文件夹完整复制到目标项目根目录，无需重复配置即可继承所有规则。
2. 新增自定义规则：在目标项目的 `.kiro/steering/` 目录下添加新的 `md` 格式规则文件，即可扩展新的 Prompt 逻辑，复用方式与现有规则完全一致。
